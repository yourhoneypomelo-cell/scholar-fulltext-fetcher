## 一、先把「本项目的 CF 到底是什么」说清楚

### 1.1 Cloudflare 2026 的三层检测（决定了「谁能过」）

2026 的横评（ianlpaterson 31 靶 651 判定）把 CF 拦截拆成**三个正交层**，绝大多数开源工具只解决前两层：

| 检测层 | 检查什么 | 谁能破 | 代理能救吗 |
|---|---|---|---|
| **① TLS/JA3/JA4 + HTTP2 指纹** | 握手密码套件顺序、H2 SETTINGS 帧序 | `curl_cffi`(impersonate) / 真浏览器 | ❌ 代理只改 IP |
| **② JS 运行时指纹** | `navigator.webdriver`、canvas/WebGL、屏幕几何 | 隐身浏览器（camoufox/patchright/nodriver） | ❌ |
| **③ automation-protocol 指纹**（关键分水岭） | **驱动浏览器的协议形状**：Playwright 启动期的 `Runtime.enable`/`Target.setAutoAttach` 握手序列 | **唯 nodriver/zendriver**（直连 CDP、无 Playwright shim） | ❌ |
| IP/ASN 信誉 | 数据中心 vs 住宅 | 住宅/移动代理 | ✅（**仅此层**） |

> **一句话**：Playwright 系（含 patchright、rebrowser、camoufox、Scrapling 的 StealthyFetcher）都栽在第③层——补 fingerprint 补不到协议层。**nodriver 因为把 Playwright 从控制面里拿掉，是当前唯一稳过第③层的开源方案**（实测唯一过 `canadianinsider` 这张 CF-Turnstile 硬盾）。**本项目 route-B 恰好用的就是 nodriver。**

### 1.2 本项目 4 个 CF 桶的画像（结合代码与既有实测）

| 桶 | Host | CF 类型 | 是否绑 JA3 | 现有路线（代码事实） |
|---|---|---|:--:|---|
| **ACS 128** | `pubs.acs.org` | managed challenge(403) | **否**（`cf_clearance` 不绑 JA3） | shim `solve()` 拿 clearance+UA → `curl_cffi` 回放**可下 PDF**（-145 实测 4.41MB）。⚠️业务侧 ~93% 是 SI，净增小 |

> **要害区分（本项目 N.3 老结论，横评再次印证）**：「**能不能过 CF 页**」与「**能不能下到 PDF 字节**」是两件事。`curl_cffi` 裸 HTTP 在横评里都能过 26/31 靶——但**绑 JA3 的站**，浏览器 solve 出的 `cf_clearance` 交给 `curl_cffi` 回放时 JA3 出口不一致 → 仍 403。故绑 JA3 站必须**同一浏览器会话内抓字节**（route-B 的 B1 页内 fetch / A Network 域 / C Fetch 拦截），这正是 `render_download_pdf_bytes` 在做的事。

## 三、权威基准（ianlpaterson 2026：7 工具 × 31 CF 靶 × 3 轮 = 651 判定）

| 浏览器 | OK | Gated | **Blocked** | 引擎 |
|---|:--:|:--:|:--:|---|
| **nodriver** | **28** | 3 | **0** ✅ | 系统 Chrome 148（直 CDP） |
| CloakBrowser | 26 | 3 | 2 | Chromium 145（49 处 C++ 补丁） |
| **curl_cffi** | 26 | 3 | 2 | curl_cffi 0.15 impersonate=chrome |
| patchright | 25 | 3 | 3 | Chrome 148 channel=chrome |
| camoufox | 25 | 3 | 3 | Firefox 135 |
| vanilla playwright | 24 | 2 | 5 | Chromium 147 |
| rebrowser-playwright | 24 | 2 | 5 | Chromium 136（已弃维护） |

**四条对本项目直接有用的结论**：
- **nodriver 唯一 0-blocked**，唯一过 `canadianinsider`（CF-Turnstile 硬盾）——**印证本项目 route-B 选 nodriver 是对的**。
- **`curl_cffi` 裸 HTTP 26/31 = 130MB 的 CloakBrowser 同分**——**印证本项目「不绑 JA3 站用 curl_cffi 回放」的轻量路线**，能省浏览器就省。
- **patchright/camoufox/Scrapling(StealthyFetcher) 都属 Playwright/Firefox 驱动**，栽在 automation-protocol 层——**不建议为过硬 CF 引入它们替换 nodriver**。
- **代理只改 IP**：Linux 服务器 + 住宅代理会制造「Linux 形状浏览器 + 住宅 IP」的矛盾指纹，**比不加代理的真机还容易被硬 block**。→ **JS 渲染类抓取（route-B）必须让浏览器进程跑在拥有住宅/干净 IP 的主机本地。**

## 四、Top 1-3 + 推荐加装件

### 🥇 Top 1：`zendriver`（= 现用 nodriver 的平滑升级） —— 最低成本、最高确定性
- **为什么**：本项目已在 nodriver 上撞到并已 monkey-patch 绕开的两类坑（`ClientSecurityState` 的 `localNetworkAccessRequestPolicy` CDP schema 漂移、偶发 `Event loop is closed`），**zendriver 已在上游修掉**（v0.15.3 修 Chrome 146 cookie `sameParty` 缺失崩溃、WebRTC 泄漏、React 输入等），并新增 **headless 下过 CF JS 挑战的 `user_agent` 选项**、**官方 Docker 模板**（真 GPU Chrome、非 headless）。
- **代价**：AGPL-3.0（内部批量回收/研究无碍；对外服务须开源或过法务）；asyncio 对象模型与 nodriver 几乎一致，迁移平滑。
- **落地**：先在 route-B 抓字节路径用小样本 A/B（zendriver vs nodriver 通过率/字节成功率），通过率不降即切。

### 🥈 Top 2：`curl_cffi`（现用，继续压榨） —— 不绑 JA3 站的轻量主力
- **为什么**：对 **ACS/AIP/T&F**（不绑 JA3）这类桶，「浏览器 solve 一次拿 `cf_clearance`+UA → curl_cffi 带 TLS 回放直下 PDF」是**最省资源**的路径（-145 ACS 已端到端实测 4.41MB）。裸 HTTP 26/31 的横评分印证其 TLS 层足够强。
- **要点**：`impersonate="chrome"`（用泛别名跟随最新指纹、别 pin `chrome124` 免 rot）；回放**必须带 solve 返回的同一 UA**，别自换 UA；v0.15.1+ 可不升库热更新指纹。

### 🥉 Top 3：`FlareSolverr /v1` 协议 + **免 Docker nodriver shim**（现用，底层要选对）
- **为什么**：`/v1` 是事实标准接口，本项目 `fulltext_fetcher/flaresolverr.py` 已逐字段兼容，且已有 `tools/flaresolverr_nodriver.py` 用 **nodriver 真 Chrome** 实现该端点——**这比官方 FlareSolverr（底层 Selenium+uc，automation-protocol 层弱）更强**。
- **更正一个流言**：官方 FlareSolverr **并未死**（14.2k star，v3.5.0 2026-05 新增 Turnstile 求解），但：① 底层 uc 对硬 CF/Turnstile 成功率下降；② 内置 CAPTCHA solver **全部失效**（README 明示）。→ **本项目继续用「FS 协议 + 自建 nodriver/zendriver shim」，不要回退到官方 uc 底。**

### ➕ 推荐加装：`nodriver-cf-verify`（处理 Turnstile checkbox 的即插件）
- 轻量 async 扩展，**同时兼容 nodriver 与 zendriver**，用 **DOM 状态判定 + CDP 坐标点击**处理 Turnstile 交互 widget，**不需 OpenCV**（比 nodriver 内置 `tab.cf_verify()` 的 opencv 视觉方案更轻）。命中 Turnstile checkbox 类质询时按需触发，接入本项目 `_nodriver_capture_fn` 的过盾等待循环几行即可。

> **备选打码（仅当业务允许且确有 Turnstile 交互谜题时）**：`flaresolverr-go` 可外接 2Captcha/CapSolver 注入 `turnstile_token`。**但对本项目当前 4 桶（managed challenge 为主，非交互谜题）非必需，且打码平台对「坏 site_key」无效（见 -165 RSC）**。

### 5.2 按改动小→收益大（排序）
1. **（0 改动）验证 AIP/T&F 是否绑 JA3**：拿 1–2 条 DOI 跑 shim solve→curl_cffi 回放，看 200/403，决定是否入 route-B 名单。**这是本波 ROI 最高的一步。**

4. **（中）住宅 IP 出口**：规模化过 Turnstile/managed challenge 的硬门槛；**且浏览器进程要在拥有该 IP 的主机本地**（别用 Linux VPS+住宅代理，会更糟）。

6. **不做**：换 Scrapling/patchright/camoufox 替换 nodriver（第③层更弱）；用 cloudscraper/cfscrape/uc（死/被取代）。

### 6.1 nodriver → zendriver（引擎工厂加一档，保留 nodriver 兜底）

```python
# fulltext_fetcher/render_fetch.py —— 新增（示意）
def _zendriver_render_fn():
    try:
        import zendriver  # 未装则 None，保持“默认关闭、优雅降级”
    except ImportError:

    import asyncio

    def _render(url, timeout):
        import zendriver as zd
        async def _go():
            browser = await zd.start(headless=False)   # 有头，CF 通过率更高
            try:
                page = await browser.get(url)
                await page.sleep(1.5)
                return await page.get_content(), (await page.evaluate("location.href") or url)
            finally:
                try: await browser.stop()
                except Exception: pass
        return asyncio.run(_go())
    return _render

# 注册（zendriver 优先、nodriver 兜底），其余 route-B 逻辑（B1/A/C 抓字节）不变：
_ENGINE_FACTORIES = {"zendriver": _zendriver_render_fn,
                     "playwright": _playwright_render_fn,
                     "nodriver": _nodriver_render_fn}
```

## 七、开源救不了的边界（如实认账）

| 边界 | 为什么开源无解 | 正解 |
|---|---|---|
| **RSC 第二道 governor** | `Invalid domain for site key` 的坏 reCAPTCHA，真人/打码平台都解不了（-165 定论） | 只能「不触发」：限速 + landing 预热 + 住宅 IP + 会话复用 |
| **数据中心 IP 过 Turnstile/managed challenge** | `cf_clearance` 绑 IP；且 TLS/HTTP2/navigator/canvas 都出自本机，代理只改 IP → 矛盾指纹更易被 block | 浏览器进程跑在**拥有住宅/干净 IP 的主机本地**（非 VPS+代理） |
| **纯订阅付费墙**（ACS/RSC 订阅刊真 403） | 不是 CF，是付费墙 | A5 机构订阅 / 商业 ScholarAPI |
| **Turnstile 交互谜题**（真图形谜题） | 开源不自解 | 外接打码（2Captcha/CapSolver），但对「坏 site_key」无效 |
| **FS 成功 ≠ 正确全文** | 求解器会忠实下载「错候选」 | 下载层加 DOI/标题内容闸门（本项目 QC 已在做） |
