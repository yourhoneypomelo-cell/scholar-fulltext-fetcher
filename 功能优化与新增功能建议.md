# 功能优化与新增功能建议(谷歌学术人机认证场景)

> 面向工作组目标的改进路线:在合规前提下,把"输入标题/DOI → 直取 PDF → 一键批量"的成功率、覆盖面与工程体验做到更高。
> 版本 1.0 · 2026-07-01 · 谷歌学术人机认证-141(总指挥)整理 · 配套阅读:`项目结构与功能介绍.md`(v2.0)、`检索成果-00-聚合总报告与选型决策.md`

## 〇、背景与目标对齐(先讲清"该往哪优化")

**工作组原始目标**:写一个专用程序,输入**标题 + DOI** → 自动完成/绕开谷歌学术人机验证 → **直取 PDF**(若有)→ 全程程序化、一键批量下载。

**当前实现(fulltext_fetcher)与目标的关系**:

| 目标拆解 | 当前状态 | 差距 |
|---|---|---|
| 输入标题/DOI | ✅ 支持 DOI/标题/arXiv/URL,标题自动 Crossref 反查 DOI | 缺 Excel/CSV 批量入口(验证用的 xlsx 需手工转 txt) |
| 过/绕谷歌学术人机验证 | ⚠️ 走**开放 API 正门**(根本不碰 Scholar、无验证码) | 没有 Scholar 原生口径(被引/版本);确需直抓时只有 PoC 骨架 |
| 直取 PDF | ✅ 22 源回退 + 校验下载 | 订阅刊语料**成功率 10.8%**(免费源天花板);"定位到却下不动"248 条多为出版商 403 |
| 全程程序化 / 一键批量 | ✅ CLI + API + 子进程 JSON + 断点续跑 + 结构化日志 | 缺图形界面/报表面板、Zotero 直写、失败分类重试 |

**总原则(本建议的取舍依据)**:① **合规优先**——能用开放 API 就不抓 Scholar;② **先榨干免费 OA**(投入小、合规、见效快);③ **再补 Scholar 原生口径**(按需、分商业/自建两条路);④ **付费墙内全文靠机构订阅**(这是突破 10.8% 天花板的唯一根本途径,非技术能绕)。

每条建议标注:**优先级**(P0 立即 / P1 近期 / P2 中长期)、**预期收益**、**工作量**、**合规风险**。

## 一、提升全文获取成功率(直接对齐"直取 PDF")

> 当前 498 条订阅刊语料:成功 54、定位到候选 302、"定位到却下不动"248(多为 403 付费墙)。这一组优化目标是回收"定位到却下不动"的那部分,以及补更多免费源。

### A1【P0】集成 CORE API key + 补绿色 OA 仓储
- **现状**:`sources/aggregators.py` 的 `Core` 连接器已写好,但无 key 时直接 `return []`(本次验证 core 命中 0/453);BASE、OSF、SSRN、Preprints.org 等绿色 OA 仓储尚未接入。
- **建议**:申请 CORE key(免费)接入;新增 BASE / OSF / Zenodo 增强 / ScienceOpen 连接器(沿用 `@register` 插件式,加类即可)。
- **预期收益**:CORE 收录 3600 万+ 全文,对绿色 OA(作者自存稿)回收明显;订阅刊语料预计 +2~4 个百分点。
- **工作量**:小(每源 ~30 行,有现成模板)。**合规**:✅ 全部开放获取。

### A2【P0】落地页解析增强(扩展 `landing.py` 规则库)
- **现状**:`landing.py` 已能抽 `citation_pdf_url` 与 `.pdf`/`/pdf` 链接,80 条样本验证 11.2%→13.8%。但对各大出版商 JS 渲染页、JSON-LD、`data-*` 属性的 PDF 直链尚未覆盖。
- **建议**:为 Elsevier(sciencedirect)、Springer(link)、Wiley、ACS、RSC、IEEE、MDPI 增加**出版商专属 selector**;补 JSON-LD(`<script type=application/ld+json>` 的 `encoding.contentUrl`)、`<meta property=citation_pdf_url>` 多变体、重定向链跟踪。
- **预期收益**:回收"定位到 HTML 落地页却下不动"的一部分(当前主要失败类型之一);保守 +2~5pp。
- **工作量**:中。**合规**:✅ 只解析已合法获取的 OA 落地页。
- **进度**:✅ **本次已落地**——`landing.py` 新增 JSON-LD(`encodingFormat=application/pdf` 的 `contentUrl/url/downloadUrl`,支持相对路径)+ `<link type=application/pdf>` 解析 + 多桶置信度排序(meta > JSON-LD > link > a/embed/iframe),selftest 扩展至 10 项全过;出版商专属 selector 与重定向链跟踪仍待补。

### A3【P1】出版商适配器层(`PublisherAdapter`)
- **现状**:各源平等遍历,未利用"DOI 前缀→出版商"的强先验(验证显示 Springer Nature 37%、Elsevier 仅 7.5%,差异巨大)。
- **建议**:新增按 DOI 前缀路由的适配器(`10.1016`→Elsevier、`10.1021`→ACS、`10.1007`→Springer…),封装各社的 TDM(Text & Data Mining)协议、Crossref `link[].URL` 的正确取法、必要的 `Accept: application/pdf` 头。
- **预期收益**:把"候选命中率高但下载成功率低"(如 Crossref 47.9% 命中却仅成功 3)的落差补上一部分。
- **工作量**:中。**合规**:✅ TDM 多为出版商对订阅机构开放的合法通道(需机构资格)。

### A4【P1】浏览器渲染兜底(可选开关,默认关)
- **现状**:`http_client` 仅 `requests`,拿不到 JS 动态渲染/Cloudflare 盾后的真实 PDF 链接。
- **建议**:新增可选 `--render` 开关,用 Playwright / nodriver 对少量"落地页解析失败"的条目做无头渲染取最终 PDF;严格限流、默认关闭、附合规提示。
- **预期收益**:回收 JS 渲染类落地页;但速度慢、依赖重,仅作兜底。
- **工作量**:中-大。**合规**:⚠️ 渲染出版商页处灰色地带,需提示;不用于抓 Scholar。

### A5【P1】机构订阅 / EZproxy / Shibboleth 集成 ★突破天花板的根本途径
- **现状**:**10.8% 是免费源对订阅刊的真实天花板**,纯订阅论文免费源拿不到属正确边界。
- **建议**:支持配置机构代理(EZproxy/WAYF/Shibboleth 登录态 cookie),下载阶段走机构通道——**这是唯一能合法拿到付费墙内全文的途径**。
- **预期收益**:对有机构订阅的用户,成功率可从 10.8% 跃升至接近订阅覆盖率(常 70%+)。
- **工作量**:中。**合规**:✅ 凭机构授权访问,合法;❗须用户自备机构账号。

## 二、补齐 Scholar 原生能力(对齐"谷歌学术检索 / 人机验证")

> 当前主线刻意"绕开 Scholar"。若用户确需 **Scholar 原生口径**(被引数、版本、相关文章)或确需直抓,这里给出分层方案。研究依据见 `检索成果-角度1/3/4/6`、`poc/self_built_stack/`。

### B1【P1】Scholar 元数据补充(被引/版本)
- **现状**:开放 API 缺 Google Scholar 口径的被引数与"所有版本"。
- **建议**:① 商业合规路线——集成 **SerpApi Google Scholar API**(付费,带法律盾,无需自己过验证码),作为可选源补被引/版本;② 轻量路线——用 Semantic Scholar / OpenAlex 的被引数近似替代(已有 API)。
- **预期收益**:满足"要 Scholar 口径"的需求,不必自建反爬。
- **工作量**:小(SerpApi 是现成 API)。**合规**:✅ SerpApi 合规;OpenAlex/S2 被引免费。

### B2【P2】反爬技术栈骨架产品化(确需直抓 Scholar 时,默认关闭)
- **现状**:`poc/self_built_stack/` 已有 scholarly + 反爬位 + 代理位骨架(dry-run、强合规警告)。
- **建议**:按 `检索成果-角度3` 的分层栈(curl_cffi TLS 指纹 → nodriver 无头浏览器 → 代理轮换 → 打码服务)产品化为**可选模块**,分层降级,默认禁用并显式合规警告。
- **预期收益**:在万不得已、用户自担风险时具备直抓 Scholar 能力。
- **工作量**:大。**合规**:🔴 高风险——违反 Scholar ToS,默认关闭、风险自负。

### B3【P2】代理池集成(为 B2 与高频 API 提供底座)
- **现状**:`检索成果-角度6` 已研究住宅/移动代理与自建池,但未接入程序。
- **建议**:新增代理池抽象(轮换 + 健康检查 + 失败摘除),`http_client` 支持 `--proxy-pool`;高频开放 API 也可用它分散限速。
- **工作量**:中。**合规**:⚠️ 取决于用途;用于开放 API 合规,用于抓 Scholar 灰色。

## 三、工程化与一键批量体验(对齐"全程程序化")

### C1【P0】Excel / CSV 批量入口
- **现状**:`cli._read_input_file` 只读 txt(逐行);规模化验证用的 `title_doi_500_batch_6.xlsx` 需手工转换。
- **建议**:支持 `-f data.xlsx/.csv`,自动识别 `title`/`doi` 列;输出**带状态的结果表**(原行 + success/source/pdf_path/error),用户一眼看全。
- **预期收益**:真正"一键批量",贴合用户给的数据形态。**工作量**:小(openpyxl/csv)。
- **进度**:✅ **本次已落地**——`cli.py` 支持 `-f *.csv`(标准库,零额外依赖)与 `-f *.xlsx/.xlsm`(可选 `openpyxl`,缺失时给出友好提示),自动识别 `doi`/`title` 列(优先 doi、回退 title)、无表头则取每行首个非空单元格,且向后兼容 `.txt`;**带状态结果表导出亦已随本轮落地**(运行后自动生成 `results.csv`,详见 C6)。

### C2【P1】失败分类重试 + 智能续跑
- **现状**:断点续跑已跳过成功项;但失败项重跑会把"404/付费墙"等**永久失败**也重试一遍,浪费时间。
- **建议**:按 `error` 分类——`no-response`/`http-5xx`/`429` 值得重试,`http-403(付费墙)`/`404`/`not-pdf` 标记为"永久失败"默认不重试(`--retry-failed` 可强制)。
- **预期收益**:大批量重跑更快、更省额度。**工作量**:小。

### C3【P1】Zotero 直写(生产版移植)
- **现状**:生产版只导出(54 篇已生成 `references.bib`/`zotero.csl.json`),`poc/scholar_multi_pipeline.py` 已有 Zotero Web API 直写。
- **建议**:把 PoC 的 `--zotero-key/--zotero-library` 直写能力移植进 `fulltext_fetcher`,下载完成即可入库。**工作量**:小。

### C4【P1】异步并发(可选重构)
- **现状**:`ThreadPoolExecutor` 跨输入并发(默认 4),已够用但线程开销偏高。
- **建议**:评估 `httpx`/`aiohttp` 异步重构,提升大批量(数千条)吞吐与资源效率;保留同步 API 兼容。**工作量**:中。

### C5【P2】Web / GUI 面板
- **建议**:轻量 Web 界面——上传清单 → 实时进度条 → 结果表下载 → 失败原因下钻;非技术用户也能用。**工作量**:中-大。

### C6【P2】运行报表可视化
- **现状**:`summary.json`/`attempts.jsonl` 数据齐全,但只能人读 JSON。
- **建议**:一键把 `summary.json` 渲染成 HTML 报表(成功率、各源贡献、出版商分档、失败原因 Top),即《规模化验证报告》的自动化版。**工作量**:小。
- **进度**:✅ **本次已落地**——新增 `report.py`,pipeline 运行结束自动生成 `report.html`(总览卡片 + 各源贡献 + 失败原因分布 + 结果明细,自包含零依赖)与 `results.csv`(带状态结果表,UTF-8-SIG)。

## 七、关键决策与边界提醒(给决策者)

1. **Scholar 直抓 vs 开放 API**:开放 API 是**唯一白色合规路线**(无验证码、官方鼓励),且已实测可跑;直抓 Scholar(B2)是灰色高风险,**仅在确需 Scholar 原生口径且愿担风险时**才启用。强烈建议主线维持"绕开 Scholar"。
2. **付费墙是物理边界,不是 bug**:10.8% 的天花板源于语料以纯订阅刊为主;**任何免费技术路线都拿不到付费墙内全文**,根本解法是机构订阅(A5)或商业解锁服务,不应靠"更强反爬"强求。
3. **被引/版本 ≠ 全文**:若用户要的是"Scholar 的被引数/版本",用 SerpApi(B1)即可,无需自建反爬抓全站。
4. **合规底线**:所有灰色路线(渲染兜底、反爬栈、代理用于 Scholar、Sci-Hub)一律默认关闭 + 显式风险提示,使用者自负。
