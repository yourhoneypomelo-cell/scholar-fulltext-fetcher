## 二、开箱就绪缺口清单

### P0 · 挡路（新用户会走错路 / 生产大面积失败）

#### ① 根 `README.md` 无一键正门入口，把新用户导向 `poc/` 原型 【最大缺口】

- **现象**：根 `README.md` 是「谷歌学术过人机认证·多角度检索」的**选型决策书**：
  - 「怎么读」(`README.md:17-23`) 指向 `检索成果-00` 主报告；
  - 「文件清单」(`README.md:26-44`) **无** `run_all.py` / `fulltext_fetcher` / 用户Runbook 任何条目；
  - 「poc 快速上手」(`README.md:48-71`) 教的是 `poc/openalex_oa_pipeline.py` / `poc/scholar_multi_pipeline.py`（**原型**），而非生产正门 `run_all.py`。
- **影响**：新用户 clone 后打开 README（第一落点）→ 只会找到 poc 原型，**根本发现不了**《用户Runbook-一键正门批量下载.md》与 `run_all.py`。这与本任务「提供标题+DOI 一键批量下载」的北极星**直接冲突**。且 `pyproject.toml:12` `readme = "README.md"` 使 PyPI/构建展示的也是这份选型书。
- **证据**：`README.md`（全篇无 run_all）、`pyproject.toml:12`。
- **建议**（属主改，本文不改）：README 顶部加一节「▶ 生产一键批量下载（正门）」直链《用户Runbook-一键正门批量下载.md》，一句话给出：
  ```powershell
  pip install -e ".[qc]"; python run_all.py -f my_papers.txt --email you@uni.edu -o out/my_batch
  ```

#### ② `[qc]` 非默认安装，但内容 QC 默认 fail-closed → 只装 `-e .` 生产近乎全 miss

- **现象**：`config.py:96` `content_qc_require_deps=True`（默认）：非 DOI-keyed 源（websearch/wayback/route-B）**需过门却缺 pypdf** 时**拒收不落盘**（`error=content-qc-deps-missing`）。而 `pip install -e .`（根 README §poc 教的 `pip install -r requirements.txt` 仅 `requests`）**不含** `pypdf`（在 `[qc]` extra，`pyproject.toml:36`）。
- **影响**：新用户若不看 Runbook、按根 README 装最小依赖跑真实批量 → websearch 等兜底源系统性拒收，**净成功率被误判为极低**、以为程序坏了。
- **证据**：`config.py:87-96`、`pyproject.toml:27-36`、`fulltext_fetcher/requirements.txt`（仅 requests）。与《依赖环境就绪核对-141.md》缺口③同源。
- **建议**：任何面向生产的开箱步骤（README + Runbook §0/§1.2）都以 `pip install -e ".[qc]"` 为**默认**，并一句话点明「省了 `[qc]` = 兜底源全被 QC 拒」。Runbook §0/§1.2 已做到 ✅；**缺的是根 README**（见①）。

### P1 · 文档-代码不一致 / 静默失效

#### ③ Runbook 写 `CORE_API_KEY`，代码实际读 `CORE_KEY` 【实锤 bug】

- **现象**：《用户Runbook-一键正门批量下载.md》§1.3 环境变量表写 **`CORE_API_KEY`**（Runbook 第 64 行），但 `cli.py:130` 实际读的是 `os.environ.get("CORE_KEY")`。
- **影响**：新用户照 Runbook 设 `$env:CORE_API_KEY` → **不生效**，CORE 聚合源静默空跑（无报错，最难排查）。CORE 为可选源，影响 P1 而非 P0，但属**照做即错**。
- **证据**：`fulltext_fetcher/cli.py:130`（`--core-key` ← `CORE_KEY`）、`config.py:44`（字段 `core_key`）、《依赖环境就绪核对-141.md》§2.2（同写 `CORE_KEY`，与代码一致）。
- **建议**：Runbook §1.3 `CORE_API_KEY` → 改 `CORE_KEY`（与代码 / 依赖核对-141 对齐）。

#### ④ `nodriver` / `curl_cffi` 未进 `pyproject` extras，开箱无 extra 可装

- **现象**：route-B（RSC/ACS/Wiley/MDPI 强 CF）与 CF 增强依赖 `nodriver` / `curl_cffi`，但 `pyproject.toml:27-36` 只有 `xlsx` / `render` / `qc` 三个 extra，**无** browser/cf。用户只能凭文档手动 `pip install nodriver curl_cffi`。
- **影响**：`--route-b cf-only|all` 在缺依赖时优雅 no-op，但新用户不会知道要手动补装。开箱视角：Runbook §3.1 的 `--route-b` 表格行仅备注「需 nodriver + 有头浏览器」，**未在命令旁给出 `pip install nodriver` 前置**。
- **证据**：`pyproject.toml:27-36`；与《依赖环境就绪核对-141.md》缺口①②同源（不重复其逐包分析）。
- **建议**：属主给 `pyproject` 加 `[browser]=["nodriver"]` / `[cf]=["curl_cffi"]` / `[full]` 聚合 extra；在此之前，Runbook §3.1 route-b 行补一句前置 `pip install nodriver`。

### P2 · 体验 / 一致性（非阻塞）

#### ⑤ `run_all.py` 无 `console_scripts` 入口，须 `cd` 仓库根跑

- **现象**：`pyproject.toml:38-40` `[project.scripts]` 只注册 `fulltext-fetcher = cli:main`，**无** `run-all`。`pip install` 后无全局命令，必须 `cd` 到仓库根 `python run_all.py`。
- **影响**：非硬缺口（Runbook 所有示例都带 `cd`），但削弱「一键」体感；非 editable 安装场景下 `run_all.py` 不随包分发。
- **建议**：属主可加 `run-all = "run_all:run"`（`run_all.py:213` 的 `run()` 已返回 int，天然适配 console_scripts）。

#### ⑥ `run_all_selftests.py` 全回归慢 + 中途无输出，易被误判卡死

- **现象**：47 项逐个 `python -m <mod>` 子进程、每个 180s 超时；本次总耗时 **~377s，其中 `render_fetch` 168.4s + `institutional` 103.1s 两项即占约 72%**（内部含浏览器启动/超时等待，离线下优雅降级仍 PASS）；子进程 stdout 非 tty 被**块缓冲**，中途终端无进度。
- **影响**：Runbook §1.2 标它「可选」正确，但用户跑时会以为卡死而 Ctrl-C。
- **建议**：Runbook §1.2 该行补注「（40+ 模块全回归，Windows 下约数分钟、中途无进度属正常，非卡死）」。

```powershell
# 1) 安装（务必带 [qc]，否则兜底源全被 QC fail-closed 拒收 —— 见缺口②）
```
