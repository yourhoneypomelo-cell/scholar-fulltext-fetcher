## 〇、TL;DR（核心发现）

**项目里"造"已基本完成，缺的是"整合"。** 存在**两套平行、成熟但未打通**的子系统：

| | 链 A · `run_all` 一键正门（现主用） | 链 B · `scholar/` 谷歌学术爬虫（独立） |
|---|---|---|
| 入口 | `python run_all.py`（一键批量） | `python -m fulltext_fetcher.scholar`（独立 CLI） |
| 标题解析 | `resolve.py`：**Crossref→OpenAlex→Semantic Scholar**（学术 API，**不走 GS**） | `fetch_serp`：**真·谷歌学术 SERP** 抓取 |
| 过反爬 | 无需（学术 API 无反爬）；下载层 route-B 过 CF | **curl_cffi→nodriver→patchright→serpapi** 分层 + captcha.py + proxy.py |
| 下载 | 多源 OA/publisher_direct/route-B | scholar-pdf 直链 + **复用父包 OA 兜底** |
| 命名 | `download.py::target_name` | `scholar/naming.py::build_filename` |
| 产物 | `out/<run>/`（coverage/still_missing/一页总结/TSV） | `out_scholar/`（serp.jsonl/attempts.jsonl/summary/csv/html） |
| 成熟度 | 成熟（本波审计≈85%达标） | **成熟**（有 SCHOLAR_PIPELINE_OK/SCHOLAR_E2E_OK 自检、断点续跑、结构化日志） |

**关键结论**：用户要的「**标题→谷歌学术爬虫→元数据+PDF 链接→…**」当前**只在链 B（独立 CLI）里存在**；而**一键正门（链 A）根本不调用 GS 爬虫**，它用学术 API 解析标题。**两链未整合**是端到端目标的**头号差距**——GS 爬虫已建好，只是没接进一键正门。

**演进主张**：**不要**用 GS 替换学术 API 主解析（后者更稳、零反爬）；而是把 **GS 爬虫作为「可选 resolver/source tier」融入 `run_all` 主环**（`--scholar off/fallback/only`），既满足"GS 在环"，又保稳健，且**复用两链所有现成件**。

## 二、逐项差距评估

### ① 输入层：支持「标题」吗？
- **支持。** 两链都经父包 `cli._read_input_file`（.txt/.csv/.xlsx）+ 标题/DOI/arXiv 混排；链 A `resolve.classify_input`、链 B `query.build_query` 均能判类。
- **差距**：标题在**链 A 被送去学术 API 反查 DOI**，**不进 GS**；用户若期望"标题→GS 结果"，当前只有链 B 独立 CLI 能给。**属"路由"差距，非"输入能力"差距。**

### ② 谷歌学术爬虫入口：有吗？若无需新增什么？
- **有，且成熟**——`scholar/` 子系统即完整 GS 爬虫（SERP 抓取+解析+选最相关+cited-by+PDF 链接+下载+标准化命名+结构化日志）。
- **差距 = 未接入一键正门**：无 `run_all --scholar`，无「标题→GS→(元数据+PDF链接)→喂给主下载」的桥接层。**需新增：run_all 开关 + resolve 层 GS 适配器**（把 `ScholarFetchResult` 适配成主 `Paper`+候选 PDF 链接）。

### ③ 过人机验证层：GS 反爬如何集成检索到的开源方案？
- **集成点已在 `scholar/fetcher.py`**：`fetch_serp` 分层 `curl_cffi(TLS impersonate)→nodriver(CDP)→patchright→serpapi(商业合规正门)` + `captcha.py` + `proxy.py`。
- **直接对接《过认证墙开源方案总评估-147》**：
  - **CAPTCHA/CF**：`NodriverEngine` → 升级 **zendriver**（修 Chrome146 cookie+CDP 漂移+headless）、加 **nodriver-cf-verify**（Turnstile checkbox）；`CurlCffiEngine` 的 `impersonate` 已在。
  - **IP 封**：`scholar/proxy.py` 挂**住宅 IP**（承接《代理编排-147》`ProxyLease` + IP-会话亲和）。
  - **避开反爬正门**：`--mode serpapi`（有 SerpApi key 走商业合规，**根本不碰 GS 反爬**）——**这是 GS 场景的"合规正门"首选**。
- **差距**：升级项（zendriver/cf-verify/住宅 IP）尚未落地；serpapi 需 key。

### ④ PDF 获取 + 文件名标准化 + 一键批量：现状与缺口
- **PDF 获取**：链 A 多源+route-B（强）、链 B scholar-pdf 直链+OA 兜底（复用父包），**都成熟**。
- **标准化命名**：**两套并存**——`download.py::target_name`（DOI→`10.x_y.pdf`/arxiv/标题[:80]）与 `scholar/naming.py::build_filename`。**缺口=未统一**：同一 DOI 经两链可能产两种命名/两处落盘。
- **一键批量**：链 A `run_all`（跨批去重/续跑/coverage/一页总结/TSV，强）；链 B `ScholarPipeline.run`（断点续跑/summary/csv/html，串行强合规）。**缺口=两个批量器、两处产物（`out/` vs `out_scholar/`）、coverage 只认链 A**——GS 抓的 PDF 不进主 coverage/still_missing/净成功率。

## 三、关键架构判断（先定方向再排模块）

**判断 1｜GS 该不该进主环？→ 作「可选 tier」，不作「主解析」。**
- 学术 API 解析（Crossref/OpenAlex/S2）**更稳、零反爬、直出干净 DOI**；GS 抓取带 CAPTCHA/IP 封风险。**主解析应维持学术 API**。
- GS 的**不可替代价值**：① 标题在 Crossref/OpenAlex/S2 查不到时的兜底；② GS 索引到的**仓库/绿 OA PDF 直链**（主链未覆盖）；③ cited-by 等 GS 专有信号。→ **GS 定位为「学术 API miss 后的 fallback + 可强制的 only 模式」**，兼顾用户"GS 在环"诉求与稳健。

**判断 2｜差距性质 = 整合 + 统一，不是从零造。** 两链的抓取/下载/命名/日志/自检都已成熟；工作量集中在**桥接入口 + 统一产物口径 + 过验证升级**。

**落地顺序**：**P1 整合入口（把 GS 接进一键正门作 fallback）→ P2 统一命名/coverage → P3 过验证升级（zendriver/住宅 IP，与总评估-147 合流）→ P4 可选**。P1 完成即兑现用户"标题→GS 爬虫→元数据+PDF→标准化→一键批量、读日志判断"的端到端形态。

**与四大现有支柱的关系（都复用、极少重写）**：
- `run_all.py`：加 `--scholar` 开关 + 桥接调用（仍是包装器，不改核心逻辑）。
- `resolve.py`：加 GS resolver tier（`title_to_doi` 全 miss 时的第 4 层）。
- `scholar/pipeline.py`：复用；可加"解析-only 返回候选"模式供 run_all 调。
- `pipeline.py` / `download.py` / `render_fetch.py`：消费 GS 候选 PDF 链接；强 CF 站走 route-B；统一命名。
- `build_coverage.py`：纳入 GS 产物，统一净口径。

## 六、诚实结论

1. **端到端目标的零件已基本造齐且成熟**（链 A 一键正门 + 链 B GS 爬虫），差距是**整合（P1 把 GS 接进一键正门）+ 统一（P2 命名/coverage）+ 过验证升级（P3 承接总评估-147）**，**非从零造**。
2. **建议 GS 作可选 fallback/only tier、学术 API 仍主解析**——兼顾用户"GS 在环"与稳健、零反爬。
3. **P1 是解锁用户目标形态的关键一步**，工作量中等；完成后"标题/DOI→GS 爬虫→过验证→元数据+PDF→标准化→一键批量+读日志"端到端闭环即成立。
