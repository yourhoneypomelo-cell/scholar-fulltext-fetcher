## 三、RSC 专项裁决（本文核心）

### 3.1 任务口径 vs 实证：RSC 不是 passive-JA3 门

任务描述「实锤仅 RSC 绑 JA3 致金 OA 0/8」，需按证据澄清。**三源一致**表明 RSC 0/8 的真门**不在** passive TLS/JA3：

| 证据源 | 关键事实 |
|---|---|
| **代码 `config.py` L107-108** | 「pubs.rsc.org/RSC 整站用 Cloudflare "Just a moment" JS 质询，连 OA 文章也拦；**普通 HTTP 与 curl_cffi(TLS 伪装)都过不了，需 JS 求解器**」 |
| **代码 `config.py` L116** | 「破 JA3 绑定型强 CF（RSC/ScienceDirect/ACS/Wiley：**curl_cffi 回放仍 403**）→ route-B 浏览器内直下」 |
| **`route-B RSC 0-8 归零因-160`** | RSC 金 OA route-B **0/8**，**主因 = 应用层 `crawlprevention/governor` rate-gate**（直导航 articlepdf + 连续快跑）+ 坏 reCAPTCHA；**JA3 绑定「不适用」route-B 页内直下路径** |
| **2026 web（curl_cffi/JA4 多源）** | TLS 指纹库只破**被动 TLS 指纹**层；**无 JS 引擎，过不了 Managed Challenge/Turnstile/企业 Bot Management**；数据中心 IP 无论指纹多真都判低信任 |

**裁决**：RSC 的门是 **Managed Challenge(需 JS/浏览器) + 应用层 governor(需限速/冷却/landing 预热) + CDN viewer + 订阅墙(唯 A5)**——**没有一道是 passive-JA3 门**。故 **curl_cffi / primp / tls-client / azuretls 等任何 TLS 指纹库都救不了 RSC 金 OA 0/8**。

「JA3 绑定」只在**「浏览器解 challenge → 把 cf_clearance 交给 curl_cffi 回放」**这条路成立（回放请求的 JA3 ≠ 解 challenge 那个会话的 JA3 → 仍 403）；但连拿到**有效 cf_clearance** 都必须先用浏览器过 JS 质询。→ **TLS 指纹库对 RSC 是「错的工具」，ROI≈0。**

### 3.2 RSC「伪装配方」（诚实版：区分无效 / 探针 / 真配方）

- ❌ **无效配方（勿投工程）**：`curl_cffi(impersonate="chrome", ja3=..., akamai=...)` 直打 `pubs.rsc.org/en/content/articlepdf/...` — 预期撞 CF Just-a-moment JS 质询（`config.py` 已实证过不了）+ governor rate-gate → challenge-page / 0 字节。
- ✅ **唯一值得的 bounded 探针（证伪用，≤3 篇、只读）**：用 curl_cffi 最新 impersonate 打 RSC 金 OA **landing 页**（非 articlepdf 直链），看响应是 challenge-page 还是 200 正文。**预期 = challenge**（与三源一致）→ 坐实「RSC 需 JS/浏览器」，正式关闭 TLS 指纹方向、结题。
- ✅ **真配方（已在 160/165 定）**：`route-B` nodriver 有头真 Chrome（自然过 managed challenge，TLS/JA3 与真浏览器逐字节一致，无需外挂指纹库）+ **-165 governor 补丁**：① governor 检测→`blocked:rsc-governor`；② landing 预热（**勿直导航 articlepdf**）；③ per-host 30–90s 限速 + 命中 governor 后 5–30min 冷却；④ 去 `--disable-blink-features=AutomationControlled`；⑤ 坏 reCAPTCHA 不硬解直接冷却；⑥（规模化）住宅 IP + 会话复用。

## 四、集成示例（落到真代码；RSC 除外）

**curl_cffi 反指纹取回已在 `http_client.py` 落地**（可选、默认关、缺库优雅降级、含离线 selftest ⑩a-g）。对「被动 JA3 门、但**无** JS 质询」的 host，**零新代码**即可启用：

```powershell
# 方式A：环境变量（一次性/CI）
$env:FTF_IMPERSONATE_HTTP = "1"      # 开启 curl_cffi impersonate 取回
# 目标浏览器指纹软读 cfg.impersonate（缺省 "chrome"）
python -m fulltext_fetcher -f still_missing.txt --email you@uni.edu
```

```python
# 方式B：配置字段（编程）
cfg.impersonate_http = True          # 开关（HttpClient._resolve_impersonate_flag 识别）
cfg.impersonate = "chrome"           # 或 "safari"（每线程独立 curl_cffi Session 注入）
# HttpClient._do_get 启用时改走 curl_cffi，复用既有重试/退避/限速/熔断；缺库逐字节回退 requests
```

- **适用**：出版商/CDN 仅以 TLS/JA3/HTTP2 指纹判机器人而 403/挂断的 host（passive 门）。
- **不适用 RSC**：`config.py` 已标 curl_cffi 过不了 RSC 的 CF JS 质询 → RSC 走 route-B（`--route-b cf-only`），**勿**指望 impersonate 开关救 RSC。
- **primp A/B**：如需独立 OS 指纹或更高吞吐，可加一层「取回引擎」抽象（`curl_cffi`/`primp`），默认 curl_cffi，primp 作实验开关（约半天工作量，见 -148 §四）。

## 五、ROI 与结论

| 池 | 规模 | TLS 指纹库净增 | 根本解 |

3. **Go 系（azuretls/utls/tls-client/requests-go）不引入**：Go 栈不适配本项目，Python 侧 curl_cffi 已等效覆盖。
