## 0. 结论先行（TL;DR）

1. **更正路线图口径：D3 并非「未建」，而是「已建 60%、待补 3 处关键增量」。** 现有 `http_client.py` 已实现：按域最小间隔限速、**基于响应头的自适应【收紧】**（Crossref `X-Rate-Limit-*`，留 20% 裕度、只收紧不放宽）、**连续失败熔断器**（阈值 3、SSL 抖动不计入、本次运行内跳过）、429/5xx 指数退避且尊重 `Retry-After`。
2. **真实空白只剩 3 个「零依赖即可补」的增量**（按 ROI）：**① 退避加 Jitter（抖动）→ ② 按域 AIMD 自适应速率（从 429 学习，不止从表头）→ ③ 熔断器加 half-open 自愈（长批跑内恢复瞬断主机）**。三者都能直接在既有结构上扩展，**无需引入任何第三方库**，与本项目「极少依赖」哲学一致。
3. **异步路径（`aio.py`，C4，-168 负责）是更大的隐患**：当前只有 `asyncio.Semaphore` 有界并发，**完全没有按域限速、没有熔断**（docstring 亦自认「生产再加每域节流」）。这是 **D3 ↔ C4 跨成员真依赖**——C4 一旦转正就会用无节流的异步客户端猛打公共 OA API。已就此**知会 -168 并报备 -156**。
4. **若允许引入 1 个依赖**：**pyresilience**（2026 新出、**纯标准库零依赖**、Resilience4j 风格）在一个装饰器里同时覆盖 retry+jitter、half-open+滑窗失败率熔断、令牌桶限速、bulkhead 并发闸——**唯一同时满足「零依赖」与「全覆盖 D3」的库**；但年轻（v0.x），采纳前须锁版本+读源码，或仅当**参考实现**抄其算法。
5. **异步转正后的限速库首选 PyrateLimiter**（v4.2.0/2026-06、MIT、async 原生、多进程/Redis 后端、requests/httpx/aiohttp extras）；**pybreaker** 作为 half-open 状态机的**参考设计**。

- 仅 `asyncio.Semaphore(max_concurrency)` 做**有界并发**；`httpx.AsyncClient` 惰性建、缺库优雅降级。

D3 的「限速」与「熔断」骨架在**同步核心已相当完整**；路线图「未建」应更正为「**已建、待补 3 增量 + 异步侧待补齐**」。

## 2. 真实空白清单（按 ROI 从高到低）

| # | 空白 | 影响 | ROI | 依赖 |
|---|---|---|---|---|
| G1 | **退避无 Jitter** | 多线程对同一 host 同拍退避 → 惊群（thundering herd），退避结束瞬间齐发再次触发 429 | **高**（改动 3 行） | 零 |
| G2 | **无 AIMD 自适应速率** | 对**不回 `X-Rate-Limit-*` 头但会 429 的源**（多数出版商）只会「每请求退避」，学不到该 host 的可持续速率；健康时也不会自动提速 | **中-高** | 零 |
| G3 | **熔断无 half-open 自愈** | 长批跑（batch4/6 数百 DOI、数小时）中一个 host 抖 2 分钟即被**整轮剔除**，白丢后续可回收命中 | **中-高**（长批尤甚） | 零 |
| G4 | 熔断按「连续计数」而非「滑窗失败率」 | 偶发一次成功即清零，退化服务判不准（脆） | 中 | 零/库 |
| G5 | **异步侧（C4）无限速/无熔断** | 异步转正后对公共 OA API 不礼貌、易被封 | 中（随 C4 优先级） | 跨 -168 |
| G6 | 无自适应并发（in-flight 限制） | 并发数写死 4，不随延迟/错误自调 | 中-低（可后置） | 大改 |

## 3. 2026 算法参考（可直接借鉴）

- **退避 + Full Jitter（AWS 架构博客，业界标准）**：`sleep = random.uniform(0, min(cap, base*2**attempt))`。另有 **Decorrelated Jitter**：`sleep = min(cap, random.uniform(base, prev*3))`（吞吐略优、更防同步）。AWS 结论：**加 Jitter 是远程客户端标准做法**，实现复杂度极低、收益巨大。→ 对症 **G1**。
- **AIMD（加性增/乘性减，TCP 拥塞控制思想；AWS SDK `adaptive` 重试模式即客户端令牌桶随响应动态调速）**：成功则**加性**放宽间隔（回到地板值方向），遇 429/限速则**乘性**收紧（×2）。带 floor（配置/默认间隔）与 ceiling（如 5–10s）。→ 对症 **G2**。
- **令牌桶 / 漏桶（Token/Leaky Bucket）**：平滑限速的标准数据结构；允许突发（桶容量）又限均速。PyrateLimiter=漏桶族，pyresilience/AWS adaptive=令牌桶。→ G2/G5 若用库。
- **熔断状态机（Nygard《Release It!》）**：`CLOSED →(阈值)→ OPEN →(reset_timeout)→ HALF_OPEN →(试探成功)→ CLOSED / (失败)→ OPEN`。pybreaker 的 `fail_max + reset_timeout` 是最简参考。→ 对症 **G3**。
- **滑窗失败率 / 延迟触发（Resilience4j / fluxgate）**：以「窗口内失败率%」或「慢调用率」开路，比「连续 N 次」更稳。→ 对症 **G4**。
- **自适应并发（Netflix concurrency-limits：AIMDLimit / Vegas / Gradient2）**：用 TCP 拥塞控制自动探测在制并发上限——`gradient=RTT_noload/RTT_actual`，>1 增窗、<1 减窗；**客户端推荐纯丢包型 AIMDLimit**。Python 侧有 `aioadaptive`、`Adaptio` 等**萌芽移植**（成熟度低，观望）。→ 对症 **G6**（后置）。

## 5. 选型结论与落地建议（三条路线）

### 路线 A（**推荐·默认**）：零依赖手写 3 增量，直接扩展 `http_client.py`
最契合「极少依赖」哲学，改动小、风险低、无供应链负担。补 **G1 Full Jitter + G2 按域 AIMD + G3 half-open 自愈**（伪代码见 §6）。合计约 40–60 行，全部在既有 `_respect_rate/_maybe_adapt_rate/_note_fail/get` 结构上增量。

### 路线 B（**若愿纳 1 个依赖**）：评估 `pyresilience` 作零依赖统一件
它**纯 stdlib、零运行时依赖**，一库覆盖 retry+jitter、half-open+滑窗失败率、令牌桶、bulkhead——恰好把 G1–G5 一次补齐，且不会引入传递依赖冲突。**风险=年轻(v0.x)**：采纳前 (1) 锁定版本、(2) 通读其熔断/限速源码、(3) 先在非关键源灰度。亦可**只当参考实现**，把其算法搬进路线 A。

### 路线 C（**异步转正时·与 -168 协同**）：`PyrateLimiter` + AIMD in-flight
C4 一旦把 `aio.py` 推向生产，`asyncio.Semaphore` 不够礼貌：应加 **PyrateLimiter（async 原生按域令牌桶）** + 一个 **AIMD 在制并发限制器**（移植 Netflix concurrency-limits 思想；关注 `aioadaptive`/`Adaptio` 成熟度）。**此项须先与 -168 对齐接口**（见 §7）。

> **一句话**：同步核心走**路线 A**（马上、零依赖、高 ROI）；把 **pyresilience** 作为路线 B 的备选/参考；**路线 C** 随 C4 排期并与 -168 联调。

## 6. 给实现者的增量设计（伪代码 · 非生产改动）

> 以下为**设计草案**供实现者照抄改造，均为对现有 `http_client.py` 的**最小增量**；保持「只收紧不放宽的地板/天花板」与「SSL 不计入熔断」等既有不变量。

### G1 · Full Jitter（替换两处 `2**attempt`）
```python
import random
def _backoff(attempt: int, cap: float, base: float = 1.0) -> float:
    # AWS Full Jitter：0..min(cap, base*2**attempt) 之间均匀取样，打散惊群
    return random.uniform(0, min(cap, base * (2 ** attempt)))
# 429/5xx：无 Retry-After 时 delay = _backoff(attempt, cap=30)
# 连接错：           delay = _backoff(attempt, cap=8)
# 有 Retry-After 仍优先尊重服务端（不加 jitter）
```

### G2 · 按域 AIMD 自适应速率（扩展 `_maybe_adapt_rate` + 429 分支）
```python
_RATE_FLOOR = cfg.per_host_interval      # 地板=配置/默认，绝不低于
_RATE_CEIL  = 10.0                        # 天花板，避免无限变慢
# 成功若干次（如连续 20 次）：加性减小间隔 interval = max(floor, interval - step)
# 命中 429/限速（即便无表头）：乘性增大 interval = min(ceil, max(interval,floor) * 2)
# 现有「按表头收紧」保留为初始/硬下限；AIMD 只在 [floor, ceil] 内自调
```

### G3 · 熔断 half-open 自愈（扩展 `_host_down` 为带时间的状态）
```python
# 由 set() 升级为 dict[host] = {"state","opened_at","fails"}
RESET_TIMEOUT = 120.0   # OPEN 后冷却 120s 再试探
# get() 入口：
#   OPEN 且 now-opened_at >= RESET_TIMEOUT → 置 HALF_OPEN，放 1 个试探请求
#   OPEN 且未到冷却            → 直接跳过（现行为）
#   HALF_OPEN 试探：成功→CLOSED(清零)；失败→重新 OPEN(opened_at=now)
```

### selftest 草案（沿用本文件既有 `_Session` 脚本注入法，不联网）
```
RATE_JITTER_OK  : mock 连续 429 → 断言退避落在 [0, cap] 且不恒等 2**n
AIMD_OK         : 注入 429×k → interval 乘性升；注入成功×N → 加性降但不破 floor
HALF_OPEN_OK    : 触发 OPEN → 快进时钟过 RESET_TIMEOUT → 断言放行 1 次试探 → 成功即 CLOSED
```

## 7. 跨成员依赖与协同（真依赖 · 已当场对齐）

## 9. 附录 · Route B 去风险：pyresilience 源码通读研判（v0.4.0 · 2026-07-02）

### 9.1 确证事实（已逐一核对）
- **License = MIT**（`LICENSE` 明文，Copyright 2026 Ahsan Sheraz）→ 可自由**引入 / vendor 内嵌拷贝 / 改写**，仅需保留版权声明。
- **零运行时依赖**（`requirements.txt` 明示「No runtime dependencies」；`pyproject.toml` 无 `dependencies`，仅可选 `fast`=uvloop/orjson、`dev`）→ 纯 stdlib、无传递依赖，契合本项目「极少依赖」。
- 类型完备（`py.typed`、mypy strict）、支持 3.9–3.14、CI/ruff/pre-commit 齐备、classifier 标 Production/Stable。

### 9.2 熔断器 `_circuit_breaker.py`（通读结论：生产级 ✅）
- 状态机 CLOSED→OPEN→HALF_OPEN→CLOSED/OPEN 正确；**`time.monotonic()` 计时**（免受系统时钟跳变影响，优于 wall-clock）。
- **单锁全量保护**所有迁移、线程安全；`__slots__` 降开销；缓存枚举成员避免热路径属性查找。
- **两种触发**：默认连续计数（低开销、向后兼容）；`sliding_window_size>0` 切换为 **`deque(maxlen=N)` 滑窗失败率 + 慢调用率**（对齐 Resilience4j，`minimum_calls` 防小样本误开）。
- **并发用心**：提供 `record_success_atomic()` 显式消除「先读 `.state` 再 `record` 两次持锁」的 TOCTOU 竞态——作者懂并发细节。
- HALF_OPEN：一次失败即回 OPEN；累计 `success_threshold` 次成功才 CLOSED；另含 `reset/force_open/force_close/metrics` 运维接口。
- **对标 pybreaker**：功能≥（多了滑窗 + 慢调用 + monotonic + 原子竞态修复）。→ **G3/G4 的最佳参考实现。**

### 9.3 限速器 `_rate_limiter.py`（通读结论：算法干净 ✅，但**非按域** ⚠️）
- 令牌桶、**整数纳秒运算**（`monotonic_ns`，免浮点漂移/开销）；`RateLimiter`(sync,线程安全) + `AsyncRateLimiter`(asyncio) 双版；`max_wait` 支持阻塞等待或非阻塞拒绝。
- ⚠️ **关键局限：单一全局桶，不按 host/key 分桶。** 本项目 D3 要**按域**限速——用它需自建 `dict[host]->RateLimiter`（与现有 `_host_interval` dict 同构）。**PyrateLimiter 原生 per-key**，此处更省事。
- ⚠️ **无 AIMD**：固定速率令牌桶，不会从 429 自学速率——故即便采用 pyresilience，**G2（按域 AIMD）仍须手写**。
