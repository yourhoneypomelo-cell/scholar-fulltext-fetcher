## 〇、TL;DR（一句话推荐）

对 **rsc / CF-hard 桶**（`cf_clearance` 绑 JA3/TLS 指纹、curl_cffi 回放仍 403）：**唯一稳的免费解 = 「solve 与 download 同一浏览器会话」——用 `nodriver`（系统真 Chrome、有头）导航到「文章页」，过 CF 后经「拦截真实 PDF 资产字节」拿回（**优先页内同源 `fetch()`、其次网络层 `getResponseBody`；均非 `Page.printToPDF` 重渲染**），首字节 `%PDF` 校验后落盘。** 本仓 `render_fetch.py::render_download_pdf_bytes()` 现有实现方向**正确**，建议保留并按 §六护栏加固。L1（curl_cffi impersonate）对 JA3-hard 无解，只做**非 JA3 站**兜底；`camoufox` 仅当 `nodriver` 也被引擎级指纹挡住时**受控升级**。

> ✅ **route-B 已实证可用（-152 实测 2026-07-02，见 §七）**：真因**不是** Chrome 旧/指纹/KeyError，而是 **route-B 在过 CF 质询期开着 CDP `Fetch` 拦截**把质询卡死。**修法（免升 Chrome、免用户授权）**：过盾期不开任何 Fetch/Network 拦截 + 用 `cf_clearance` cookie 判过盾 + 过盾后走 B1 页内 fetch；KeyError（Chrome 146 PNA→LNA 改名）用 runtime monkeypatch 兜底。实测 `10.1021/acsomega.6c04195` CF **6.6s 过**、B1 抓到 **13.7MB / 17 页真 PDF、QC=match**。**Chrome 133 + nodriver 0.50.3 无需升级即可用。**

## 一、问题定性：为什么 rsc/CF-hard 是「JA3 死结」

Cloudflare 的 `cf_clearance` cookie **同时绑定** 三要素：**发起挑战的 IP + User-Agent + TLS/JA3(JA4) 指纹**（外部 2026 源与仓内 -145/-179 实测一致）。因此：

- 用浏览器过 CF 拿到 `cf_clearance` 后，把 PDF URL 交给 `curl_cffi`/`requests` **另发一个 HTTP 请求回放** → **出口 JA3 变了** → Cloudflare 判为 cookie 被盗用 → **403**（换更强求解器 byparr、换 TLS 客户端都无效）。
- 三类常见误区（均会让 cookie「一用即废」）：① 会话中途轮换代理（solve 在 IP-A、取回走 IP-B）；② 浏览器 solve 后交裸 `requests` 提交（指纹/UA 不匹配）；③ UA 谎称 Chrome 却发 Python 的 JA3（自相矛盾）。

> **根因**：瓶颈在**「回放下载」这一步换了出口指纹**，不在「HTTP 客户端够不够像浏览器」。所以**换更强的 L1 客户端解不了 JA3 死结**——只能**别换出口**：把下载留在同一个浏览器会话里。

## 二、三层技术栈定位（务必别混为一谈）

| 层 | 代表 | 解决的问题 | 对 rsc/CF-hard 的作用 |
|---|---|---|---|
| **L1 HTTP-TLS 指纹客户端** | curl_cffi / primp / tls-client | 无浏览器直连时不被 JA3 一眼判死 | ❌ 对「绑会话的 `cf_clearance` 回放」**无效**；仅救非 JA3 站 |
| **L2 隐身浏览器引擎** | nodriver / patchright / camoufox | 让浏览器**不被识别**、**过 CF 质询本身** | ✅ 必需（先过盾） |
| **L3 浏览器内抓字节（CDP）** | `Fetch`/`Network.getResponseBody` | 过盾后**不换出口地**把 PDF 拿回来 | ✅ 钥匙（JA3 死结的真正解法） |

**rsc/CF-hard = 必须 L2 + L3 合体**。L1 单独无解、L3 单独没盾可过。

## 五、L3 · 浏览器内直下 PDF：`Page.printToPDF` vs 拦截真实资产字节（本文重点 · 任务点名）

过盾之后有两条「在浏览器内拿 PDF」的路，**本质完全不同，务必选对**：

| 维度 | **A. `Page.printToPDF`** | **B. `Fetch`/`Network.getResponseBody` 拦截真实响应** |
|---|---|---|
| **本质** | 把当前 **DOM 虚拟打印**、**重渲染**成一个**新** PDF（虚拟打印机） | **截获服务器真实传输的原始文件字节** |
| **保真度** | 应用 `print` CSS、重排版式/分页/颜色 → **非出版商原版** | **100% 原样**（出版商原始排版 PDF） |
| **下载「已存在的 PDF 文件」** | ❌ **不应用于此**（二次生成、失真；官方/社区明确「不要用 printToPDF 下载既有 PDF」） | ✅ **正解** |
| **只有 HTML 全文、无 PDF 资产时** | ✅ **唯一出路**（HTML→PDF） | ❌ 无资产可截 |
| **JA3 出口** | 浏览器自身（会话内，一致） | 浏览器自身（会话内，一致） |
| **可检索文本/元数据** | 取决于渲染（可能丢矢量文字层） | **原样保留** |
| **大文件** | `data`(base64) 一次性，或 `returnAsStream` 取流 | `getResponseBody` 一次性；超大用 `takeResponseBodyAsStream` **流式**读 |
| **主要实现坑** | 需 `emulateMediaType('screen')`、`-webkit-print-color-adjust`、页边距 | 须在 **Response 阶段暂停**才能取 body；**重定向无 body**；`Fetch`+`Network` **双通道**兜底（有站走 fetch 拦截、有站走 network 完成事件） |

### 推荐（对 rsc/CF-hard 桶）

**用 B —— 拦截真实资产字节**（`Fetch.enable requestStage=Response` + `getResponseBody`，配 `Network` 完成事件双通道，首字节 `%PDF` 校验）。理由：我们要的是**出版商原始排版 PDF**（含矢量文字、可检索、正确分页），`printToPDF` 会重渲染失真、且对「浏览器内置 pdfjs viewer 打开的 PDF」重打印会丢页/丢文字层。

**`Page.printToPDF` 的定位**：**仅**当确认「该文献只有 HTML 全文、根本没有 PDF 资产可截」时，作为**可选补充**产出一份 HTML→PDF（并**另行标记来源为「重渲染版」**，内容 QC 需知情）。**不作为 rsc/CF-hard 的主路径。**

> **落点核对**：仓内 `render_fetch.py::render_download_pdf_bytes()` 已按 **B（拦截字节）** 实现（`Fetch.enable(RequestStage.RESPONSE)` → `get_response_body` → `%PDF` 校验，`Network` 完成事件双通道，`Network.enable` buffer 调大到 ~100–120MB 防大 PDF 被丢）→ **方向正确、无需返工**。可选加固：超大 PDF 评估 `takeResponseBodyAsStream` 流式，规避单串内存峰值。

### 5.1 拦截真实字节的两条子路径：B1 页内同源 `fetch` / B2 网络层 `getResponseBody`（承 -152 实测细化）

「拦截真实资产字节」实际有**两条实现子路径**，-152 的 route-B 真机验收给出了优先序：

| 子路径 | 做法 | 出口一致性 | 对 rsc 适配 |
|---|---|---|---|
| **B1（首选）· 页内同源 `fetch`** | 过 CF 后**在文章页上下文内** `fetch(pdfUrl, {credentials:'include'})` → `FileReader` 转 data-URL → `%PDF` 校验 | **同 cookie + 同 JA3 + 同源**（页内请求天然继承 `cf_clearance`+JA3） | **最稳**（Cloudflare 无从判别是「另发请求」） |
| **B2（兜底）· 网络层拦截** | 导航到 PDF 直链，`Fetch.enable(RESPONSE)` + `getResponseBody` 网络层抓 | 浏览器出口，一致；但**无 CORS 顾虑** | 兜底（仓内现即此路） |

- **B1/B2 分域适用（-152 实测细化 2026-07-02）**：**ACS-OA 等同源 PDF → B1**（页内 fetch 成功、最稳，实证 `acsomega.6c04195`）；**RSC → B2**——B1 页内 fetch `articlepdf` 报 `TypeError: Failed to fetch`（CSP/跨源限制），须走 B2（过盾后再开 `Fetch.enable(RESPONSE)`、导航到 PDF 直链在网络层抓；navigation 不受 CSP `connect-src` 约束）。两者均 `%PDF` 校验；`Page.printToPDF` 仍**不用**（重渲染失真、不利 QC）。
- **RSC 额外依赖**：RSC 文章页**无明链 PDF**，需 `publisher_direct._rsc()` 由 DOI 构造 `articlepdf/{year}/{jcode}/{suffix}` URL 作为 B2 的导航目标（该构造仓内已有）。
- **实现建议**：`render_download_pdf_bytes()` **先试 B1、遇跨源/CSP 失败自动回落 B2**；B2 的 `Fetch.enable` **只在过盾后开**，绝不在质询期开（否则卡 CF，见 §七）。

## 六、给 rsc/CF-hard 桶的可落地推荐路径（决策流）

```
CF 终态(cloudflare-challenge) 且 is_ja3_bound_cf_host(url)?
        │是                                     │否
        ▼                                       ▼
route B: render_download_pdf_bytes()      L1: curl_cffi impersonate 回放
  1) nodriver 有头 + 系统真 Chrome           (ACS 等非 JA3 站, cf_clearance 可复用)
  2) 导航「文章页」(非 md5 短链)
  3) Fetch+Network 双通道拦 %PDF 字节
     + block-page 文本检测 + deadline
  4) %PDF + %%EOF + size + 内容QC(DOI/title)
  5) 按 origin 缓存会话; 强限流; 默认 gated 关
        │仍被挡(引擎级指纹)?
        ▼
  受控升级 camoufox(稳定分支); 仍失败 → 判「订阅墙/无免费全文」
   (no-pdf 是正确行为, 非 bug)
```

- **触发点**：仅 `download.py` 检出终态 CF **且** host ∈ JA3 绑定型强 CF 站（rsc/sciencedirect…）时才走这条重路径；传**文章页 URL**、**不传 pdf 短链**。
- **诚实 ROI**（承 -173 runbook / ROI）：全量 still_missing 净增点估 **≈ +20 篇（+2pp）**；route B 验的是**机制**，不是批量覆盖率银弹。rsc 桶 67 条里多数可能是订阅墙（`no-pdf` 即正确）；真正被 route B 独有救回的样本务必存档（DOI + PDF hash）。

## 七、与 -152 route-B 实测协作（一手实测已回填）

-152 于 2026-07-02 完成 route-B 真机有头验收（详见其《路线B-真机冒烟结果-152.md》），四点一手结论：

| 问题 | -152 一手结论 |
|---|---|
| **rsc 能否真拿 %PDF** | **本机未能验证**——CF 这关就没过（title=`Just a moment...`、body=`Performing security verification`）；rsc `c1gc15503b` 实测 `no-pdf`，但那条本是订阅墙，**无法区分「订阅无 PDF」vs「CF 未过」**。0/4 拿到 %PDF |
| **引擎 / 模式 / 通过率** | `nodriver 0.50.3`（render_fetch 自带 CDP、零新依赖）+ 系统 Chrome、**有头**；**CF 通过率本机 0/4**；playwright/patchright 未测 |
| **抓字节方式** | **拦截真实响应体（非 printToPDF）**；优先 **B1 页内同源 `fetch`**（继承 `cf_clearance`+JA3、最稳）、**B2 网络层 `getResponseBody`** 兜底；`printToPDF` 得重渲版、不利存档/QC → 不用 |
| **curl_cffi 回放 403 复现** | 本轮**未专门复现**（其任务是有头验收；Runbook 第 4 步的对照实验因本机 CF 未过、未跑到）；结论引 **-145/-179** 实测 + shard roi 备注为准 |

### ⚠️ 决定性前置风险（-152 实测 + -148 溯源，P0 先解）

nodriver 0.50.3 对每条 `requestWillBeSentExtraInfo` 抛 **`KeyError: localNetworkAccessRequestPolicy`**（`network.py:2324`）。**精确根因**（-148 实读本机 nodriver 源码 + 上游核实）：Chromium 在 **Chrome 146** 把 CDP 字段 `privateNetworkRequestPolicy` 改名为 `localNetworkAccessRequestPolicy`（Private Network Access → Local Network Access 改名）；nodriver 0.50.3 的 `ClientSecurityState.from_json`（`network.py:2320-2325`）**硬取新键**，即它按 Chrome 146+ 生成 → 跑在 **Chrome 133（旧、仍发 `privateNetworkRequestPolicy`）** 上必 KeyError。**精确阈值是 146，不是 140。**

**三条修法**（已喂 -152，task-6933711b）：
1. **升 Chrome→146+**（发新键即匹配，KeyError 自消）——最省事；建议先查 -144 过 CF 那台的 Chrome 版本，若 146+ 直接对齐。
2. **免升 Chrome 的 runtime monkeypatch**：`nd.start()` 前对 `cdp.network.ClientSecurityState.from_json` 打补丁，双键兼容 + 非法值兜底 `Allow`（新枚举 `LocalNetworkAccessRequestPolicy` 仅 `Allow / BlockFromInsecureToMorePrivate / WarnFromInsecureToMorePrivate / PermissionBlock / PermissionWarn`）。
3. 参考 nodriver 活跃 fork **zendriver issue #249 / PR #250**（`handle renamed CDP policy`，双键兼容）。
4. `_BLOCK_SIGNALS` 建议补 `performing security verification` / `you are not a bot`（-152 实测质询页文案）。

> **综合判断**：本文 §五/§六 的**方法选择正确**（拦真实字节、优先页内同源 fetch、非 printToPDF），但 **rsc 端到端尚未被证实**——瓶颈在**上游「过 CF」**，当前受制于 **Chrome 133 + nodriver 0.50.3 的字段改名不匹配**（非抓字节方法本身）。**行动优先级**：P0 修 env（Chrome→146+ / monkeypatch / 钉 nodriver）→ 复跑 `路线B-…Runbook-173.md` 验 rsc 端到端 → 再谈批量回收。

> **两类问题已厘清（-144 + -152 双实测）**：① **FS 路径**——-144 实测「单干净 shim 过 CF：200 + cf_clearance@6.9s」，故 CF 桶主症是**重载争用（非 Cloudflare 真破）**，FS 降并行（单 shim 单桶串行）即可出货；② **route-B 路径**——本机 0/4 是上述 **nodriver/Chrome 字段改名 env-bug**（route-B 的字节抓取依赖 `Network`/`Fetch` 事件处理器，故此 KeyError 对 route-B 比 FS 更致命）。两者是**不同**问题、各自有解，不可混为「CF 把我们封死」。

### route-B 过盾更深层根因（-152 monkeypatch 实测后仍 0/4 → -148 对读源码定位）

-152 实测：**打了 KeyError monkeypatch（KeyError 洪水消失）后，CF 仍未过、route-B 仍 no-pdf**（浏览器停在 `Just a moment` / `Performing security verification`）。→ **修 KeyError 必要但不充分**。-148 对读 FS 与 route-B 源码，定位最可能真因：

| 维度 | FS `_solve`（过 CF ✅，含 headless） | render_fetch route-B `_capture`（本机 0/4 ❌） |
|---|---|---|
| CDP 拦截 | **全程不开**（仅 `get_content`/`get_all_cookies` 读） | **导航前即 `fetch.enable(patterns)` + `network.enable`**（质询期拦截在开） |
| 过盾判定 | **等 `cf_clearance` cookie 出现 + 质询文案消失**（2s 轮询、≥8s） | 看 `document.body.innerText` 是否含 block-signal |

**主假设**：CF 对 CDP `Fetch` 请求拦截高度敏感——route-B 在**过盾期间开着 Fetch 拦截**，很可能正是它拦住了质询通过；而 FS 纯导航 + cookie 轮询、无拦截，故同机同 Chrome 133（甚至 headless）能过。**指纹层可排除**（FS 同 Chrome 133 能过）；**启动/隐身参数亦可排除**——实读对比 FS `_boot`（`flaresolverr_nodriver.py:118-120`）与 route-B（`render_fetch.py:438-440`）的 `browser_args` 基本一致（同 `--lang=en-US` / `--disable-blink-features=AutomationControlled` / `--no-first-run` / `--no-default-browser-check`，仅窗口 1400 vs 1600）。→ **唯一材质差异就是「过盾期是否开 CDP Fetch 拦截」。**

**建议改法**（时序按 FS 流重构）：① 导航后**先不开任何拦截**，按 FS 流轮询 `get_all_cookies()` 等 `cf_clearance` 出现且质询文案消失（参考 FS `_solve` 行 185-197）；② 过盾后再抓字节——**B1 页内 `fetch` 本就不需 `Fetch.enable`**，优先走；仅 B1 失败才临时开 `Fetch.enable` 走 B2；③ 过盾判定改用 cookie（质询常在 iframe/shadow DOM，`innerText` 探不到 marker）。KeyError monkeypatch 仍保留（B2 用 Fetch 时需要）。

> ✅ **已由 -152 实测确认（2026-07-02，`_route_b_cfwait_152.py`）**：过盾期不开任何 Fetch/Network 拦截、纯导航 + 轮询 `cf_clearance`、过盾后才 B1 页内 fetch —— `10.1021/acsomega.6c04195`（ACS Omega 金 OA）**CF 6.6s 过**（`has_cf_clearance=True`）、B1 拿到 `%PDF-1.4`、**13,715,921 字节 / 17 页**、内容 **QC=match**（DOI-in-text + 标题关键词全中）。**结论：route-B 在 Chrome 133 + nodriver 0.50.3 无需升级即可用。** 一处补坑（-152）：过盾判定**别用 HTML 里的 `cf-chl` / `challenge-platform` 脚本标记**（过盾后正常页仍残留这些脚本 → 假阳），用 **`cf_clearance` cookie + 可见文案（title / body innerText）** 判。落地见 `路线B-真机冒烟结果-152.md` UPDATE 节。
