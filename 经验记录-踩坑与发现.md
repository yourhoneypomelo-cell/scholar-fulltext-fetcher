## A. 多源全文抓取的踩坑(fulltext_fetcher 实现期实测)

1. **落地页冒充 PDF(头号坑)**
   `oa_url` / `bibjson.link.url` / 部分 `best_oa_location.url` 返回的是 HTML 落地页而非 PDF。
   ✅ 对策:下载后校验 `Content-Type` 非 html + 内容前 1KB 含 `%PDF` 魔数 + 大小达标;失败记 `landing-page`/`not-pdf`。

2. **某些 host 在特定网络环境连不通会拖垮整体**
   实测本环境对 `pmc.ncbi.nlm.nih.gov` 持续 `SSLEOFError`,默认重试导致**每条输入白等 ~25s**。
   ✅ 对策:加**按域熔断器**——连续 3 次(非 SSL)连接类错误即本次运行跳过该 host(阈值原为 2,后提至 3 以减少高峰误熔)。改完总耗时 40s→7s(对命中靠前源的样本)。

3. **`--no-download` 汇报了错误的 top 候选**
   初版只在 `landing` 列表里挑"top",忽略了直链,导致 peerj 明明有 Unpaywall 8 个直链却报了个 `doi.org` 落地页。
   ✅ 对策:累积**全部**候选,按 `(is_direct, confidence)` 排序取 top。

4. **OpenAIRE 旧版 JSON 杂质极多**
   深层嵌套里递归抓 url 会抓到出版商/机构首页(如 `universityofcalifornia.edu`)这种垃圾。
   ✅ 对策:只保留含 `pdf`/`/download` 或已知 OA host 的链接,其余丢弃;OpenAIRE 仅作低优先兜底。

5. **Unpaywall 占位邮箱 422**
   `@example.com` 会被拒(422);`@example.org` 实测可用,但生产**必须填真实邮箱**。无邮箱时 Unpaywall(命中率最高的源)直接废掉。

6. **arXiv 合成 DOI 在 OpenAlex 不一定解析**
   `10.48550/arxiv.XXXX` 走 OpenAlex 单条可能拿不到元数据(title 为空)。
   ✅ 对策:arXiv 源直接用 arxiv_id 构造 `arxiv.org/pdf/{id}` 直链,不依赖元数据富化即可命中。

7. **arXiv API 限速 3s/次**
   官方要求 ≥3s/请求、单连接。对非 arXiv 论文做标题搜索会白白等 3s。
   ✅ 对策:为 `export.arxiv.org` 单独设 3s 间隔;下载模式靠"首个直链即短路",命中靠前源时根本不会触发 arXiv。

8. **作为子程序的 JSON 出口要强制 UTF-8 字节**
   `--print-json` 若用 `print()`,在 Windows 上被父进程捕获时易受控制台/区域编码影响;再叠加 PowerShell `>` 重定向会写成 **UTF-16 BOM**(`0xff 0xfe`),父程序按 utf-8 解析直接崩。
   ✅ 对策:CLI 用 `sys.stdout.buffer.write(json.encode("utf-8"))` 强制 UTF-8 字节;父程序用 `subprocess(capture_output=True)` 拿 bytes 再 `.decode("utf-8")`,绕开 shell 重定向编码。已验证可稳定 `json.loads`。

9. **Crossref link[] 是高噪声 TDM 链(候选率最高却几乎下不动)**
   实测 Crossref 候选产出率 48.6%(最高),但 `link[]` 多为出版商 text-mining 链 → 普通请求 403(30 次);是"定位到却下不动"的主因。
   ✅ 对策(2026-07-01 落地):抽 `_score_crossref_link` 纯函数给 Crossref 候选**分层降权**——基础分压到 `application/pdf=40` / `unspecified=24`(均低于真 OA 直链最低 68),对 `intended-application=text-mining/similarity-checking` 再 −12、已知订阅出版商域名(elsevier/acs/rsc/wiley/oup/sage/t&f…)再 −10,下界 5。**保留为低分候选不丢弃**(下载到落地页仍可由 landing 二次回收,实测能回收一部分)。效果:`--no-download` 定位模式 top 不再被 Crossref 兜底链抢占;下载模式源内先试更可能成功的链。`python -m fulltext_fetcher.sources.aggregators` 有 selftest 防回归。

## D. 协作 / 环境经验(PChat 工作组)

1. **PChat 工具调用别手工 `\uXXXX` 转义**:直接传中文纯文本,手工 Unicode 转义会导致 `delegate_task` 等调用失败(组内多次踩到)。

3. **公共 backlog 池可能跨组**:`list_backlog` 里出现过别的项目任务,push 前要确认归属,避免进错池。
4. **Windows PowerShell 控制台中文 GBK 乱码**:仅显示问题,落盘文件均 UTF-8,功能不受影响。
5. **CLI help / 日志里的 emoji 会在 GBK 控制台直接崩溃(不是乱码,是崩)**:argparse `--help` 打印含 `⚠️`(U+26A0)的帮助串时,Windows GBK 控制台报 `UnicodeEncodeError: 'gbk' codec can't encode character '\u26a0'` 而**整条命令失败**(`--selftest` 不打印 help,故长期未暴露;真机跑 `--help` 才发现)。⚠ 注意区别:中文本身 GBK 可编码(不崩、正常显示),**emoji 才是元凶**。✅ 对策:CLI 帮助串 / 日志告警只用 GBK 可编码字符,把 `⚠️` 换成「注意:」。已修 `cli.py`(help+warning)与 `scholar/cli.py`(help);`python -m fulltext_fetcher --help` / `... scholar --help` 恢复正常(退出 0)。(模块 docstring 里的 emoji 无妨——不会经控制台编码。)

## E. 规模化验证实测(80 真实 DOI,paywall 为主语料,2026-07-01)

数据来自 `title_doi_500_batch_6.xlsx`(Elsevier 240 / ACS 105 / RSC 52 / Springer 29 / Wiley 23 / MDPI 15…),取前 80 条 DOI 用 fulltext_fetcher 实跑(并发10、含下载)。

**结果:成功 9/80 = 11.2%**(命中源:Unpaywall 5、Europe PMC 2、OpenAIRE 1、S2 1);平均 17.8s/条;总 151.5s。

**各源候选命中率 vs 实际下载**:
| 源 | 产出候选率 | 说明 |
|---|---|---|
| Crossref | 48.6%(最高) | 但几乎都下不动:`link[]` 多为出版商 TDM 链 → **403**(30 次)。高噪声、低产出 |
| Unpaywall | 21.2% | 真正贡献 5 篇;部分是 bronze 落地页/403(10 次) |
| OpenAlex | 16.0% | — |
| Semantic Scholar | 12.3% | 贡献 1 篇 |
| Europe PMC | 2.7% | 化学语料生物医学占比低 |
| CORE | 0 | **未配 key 被跳过**(本可能补绿色 OA) |
| DOAJ/HAL/Zenodo/PMC | 0 | 该语料几乎无命中 |

**关键发现**:
1. **47/80 "定位到候选" 但仅 9 下成**——38 篇定位到却下不动,绝大多数是**出版商 403(付费墙)**,不是程序缺陷。订阅型化学语料的免费全文天花板本就低。
2. **11% 是诚实的免费源覆盖率**:Elsevier/ACS/RSC 多数无免费版,其 Crossref 链是付费 TDM。要更高只能上**角度4 商业服务**或机构订阅(正是七角度研究的结论)。
3. **可提升项**:① 配免费 **CORE key**(36M+ 全文,补绿色 OA);② 实现 **HTML 落地页内 PDF 解析**(可回收一部分 ct=text/html 的落地页);③ 真实机构邮箱可小幅提升 Unpaywall;④ Crossref 候选分层降权(高噪声)——✅ 已落地(见 A.9)。
4. **诚实边界**:付费墙内全文任何免费手段都拿不到;本程序对**开放获取/绿色 OA**的论文命中良好,对纯订阅论文返回"无免费版"是正确行为。
5. **HTML 落地页解析已落地并验证有效**:`landing.py`(144 写,提取 `citation_pdf_url`/内嵌 `.pdf` 链接)集成进 `download.py`(拿到 HTML 时抠出内嵌 PDF 再下一层、`%PDF` 二次校验)。同一 80 条样本 **9→11(11.2%→13.8%,相对 +22%)**,且 **Crossref 从 0 变 2**(落地页里的 citation_pdf_url 被成功回收)。结论:landing 解析能稳定回收一部分"定位到却返回 HTML"的失败;对硬 403 无效(那是付费墙)。

## F. 谷歌学术爬虫子系统 fulltext_fetcher/scholar/(2026-07-01 构建 + 真实验收)

> 目标:输入标题/DOI → 避免或自动过人机验证 → 抓 Scholar 元数据 + 可及 PDF → 文件名标准化 → 批量。经「研究(R1 反爬栈 / R2 Scholar抓取·CAPTCHA·下载)→ 架构与选型(冻结接口契约)→ 14 模块并行构建」完成,离线 selftest 全绿,并做了真实 Scholar 联网验收。详见 `谷歌学术爬虫-架构与选型.md` / `谷歌学术爬虫-使用说明.md`。

**架构要点**
- 三层+两翼反爬:`curl_cffi`(L1 TLS 指纹 impersonate)→ `nodriver`(L3 无头浏览器);翼A 住宅代理、翼B 限速/打码。`mode=auto`(有 SerpApi key 走合规商业,否则自建)。
- 大量复用父包(download/landing/render_fetch/sources/resolve/report/logsetup),不重造轮子。
- 默认最安全:代理/打码/Sci-Hub/OA兜底 **全关**;Scholar 抓取串行 + 强限速(页间 45–90s、命中风控冷却 900s)。

**踩坑 / 新发现**
1. **数据中心 IP 直连 Scholar 必被 403**(实测):真跑 "Attention is all you need" → 即便 curl_cffi(impersonate=chrome)也**即刻 403**。结论:TLS 指纹伪装过不了 **IP 信誉**这关;真实抓取必须住宅代理或直接上 SerpApi(合规)——印证 R1/R2 调研。
2. **403 未被判「被拦」的 bug(靠 run+读日志 当场抓到)**:`fetcher._is_blocked` 初版只认 429/503、漏了 403,而 `serp.detect_captcha` 是认 403 的——不一致导致 403 被当普通失败、跳过 冷却/换代理/升级浏览器。✅ 已修:`_is_blocked` 统一走 `detect_captcha(html,url,status)`;修后 403→blocked=True + 触发 block 事件 + 冷却。**这就是「改程序→跑→读日志→再改」闭环的价值**。
3. **离线 mock selftest 验的是「编排正确性」,不是「真实页面选择器准确性」**:`serp.py` 用合成 SERP HTML 自测(SERP_OK),但真实 Scholar 现行 HTML 的选择器(`.gs_r/.gs_rt/.gs_a/.gs_fl/.gs_or_ggsm`)**未在真实页面校验过**——因为被 403 挡着、从没拿到真实 SERP。这是**唯一未闭合、且属环境物理边界(非代码缺陷)**的差距。
4. **对策:原始 SERP HTML 快照**(ARCH §6,本轮补齐 pipeline 实现):每次取回把原始页面落 `out_scholar/serp/<idx>_<engine>_<status>.html`。在**非被拦环境**跑一次即存下真实页面,据此离线校对/调优 `serp.py` 选择器,无需反复联网。
5. **OA 兜底默认关**(用户拍板):北极星是「Scholar 所能达到的」,闭源/付费墙用户用别的渠道;故 `config.oa_fallback` 默认 False,`--oa-fallback` 显式开启。

**成功经验 / 协作**
1. **研究先行 + 先冻契约再并行**:R1/R2 调研 → 综合《架构与选型》**冻结 models/config 接口契约(P0)** → 多成员并行实现 P1–P4,几乎无接口漂移。**教训**:个别模块(serp/naming/query)在 ARCH 前抢先起步、用了 dict 接口,后需回头对齐 models,略有返工——**契约先冻结再并行**能免掉。
2. **可选依赖一律函数内延迟导入 + 统一「不可用信封」**`{available:False, reason}`:curl_cffi/nodriver/patchright/2captcha/capsolver 全不进强制依赖,缺失即优雅降级。
3. **一键回归覆盖全项目**:`run_all_selftests.py` 现含 36 项(compileall + 35 模块 selftest,其中 scholar 子系统 11 项),`python run_all_selftests.py` 一条命令验全项目(PASS=36 / FAIL=0 / SKIP=0)。

**入口**:`python -m fulltext_fetcher.scholar "标题或DOI" [-f 列表] [--serpapi-key … | --proxy]`;产物落 `out_scholar/`。

### H.1 【突破】websearch + publisher_oa 把免费公开边界成功率从 10.8% 拉到 ~82%

  - 早期"纯 OA-API 聚合"口径(仅 unpaywall/openalex/s2/crossref…,**无 websearch**):`val500_out/summary.json` 处理 498、成功 54 = **10.8%**。
  - 上线 **websearch**(DuckDuckGo/Bing 找作者自存稿·机构库·预印本)+ **publisher_oa**(OA 出版商 DOI→PDF 直构)后,**同一 500 DOI 语料**:`out/batch6/summary.json` 本轮 success=336 + skipped_resume=74(前轮已成功续跑跳过)= **累计 410/500 = 82.0%**;去重合并另 +2 = **412/500 = 82.4%**(见 `最终交付清单.md`)。
  - by_source(batch6):**websearch 260**、publisher_oa:acs-authorchoice 52、unpaywall 11、europe_pmc 7、s2 2、crossref 2 → **websearch 单源约占成功 72%**(本轮 260/336≈77%),是拉高命中的绝对主力。
- **结论**:**不依赖 SerpApi key / 住宅代理**,仅靠免费搜索引擎兜底 + OA 出版商直构,就能把"公开网络可及"的免费 PDF 命中率从个位数拉到 ~82%(订阅刊化学语料);付费墙内纯订阅全文仍是物理边界(需机构订阅 A5)。

### H.2 【坑+修复】批处理收尾被单条 straggler 挂起 → summary 不落盘(已加看门狗)
- **坑**:并发批跑收尾时,若个别输入的 future 卡在阻塞 IO(如 `10.1039/d1ta08016d` 在 websearch/落地页阶段长时间无返回),`ThreadPoolExecutor` 收尾整体挂起 → `summary.json` 迟迟不落盘,无人值守跑批"看起来死了"。
- ✅ **修复(已落地,证据 `fulltext_fetcher/pipeline.py`)**:尾部卡死看门狗——
  - `_straggler_timeout()`(L228):无进展阈值秒 = `max(180, per-request 超时×8)`(或 `cfg.item_timeout`)。
  - 主循环(L264–282)`wait(pending, timeout=straggler_timeout, return_when=FIRST_COMPLETED)`;一个窗口内**零完成**→ 判卡死,`log.error("尾部 N 条…判定卡死(straggler)")`,`cancel()` 未启动项 + `_record_straggler()`(L242)把卡死项记为 `error="straggler-timeout"` 失败结果并落盘,`break` 停等;`finally: ex.shutdown(wait=False, cancel_futures=True)` 不等卡死线程。

  - 跨社按**出版商标签**判而非裸 DOI 前缀(Wiley 10.1002/10.1111 同社多前缀不算跨社)。
- **⚠️ 交集→并集的判定教训(与 L.3 呼应,代价:一轮返工)**:第一轮实现曾疑"标题法过判"而取**交集**(标题不符 AND 跨社信号)。审计逐条交叉验证推翻:content-mismatch(标题分<50)≈100% 准,且 **189 条同域错论文(publisher 61 + repository 128:Springer DOI→另一篇 Springer 文、或仓库托管他篇)跨社信号根本不触发**——交集会整批放行。定稿**并集**,并新增「同社错论文」selftest(Springer→Springer 仅标题不符、无跨社信号 → 必判 mismatch;先断言第二信号不触发,证明交集在此必漏)锁死不回退。

## N. CF 免 Docker 破盾落地 + JA3 绑定三层拆解 + 路线B 浏览器内直下(2026-07-02 固化,146;源 -145/-173/-177/-179/-149/-150/-148)

### N.1 免 Docker 过 CF = 仓内 nodriver shim(起法 + 端口 + FLARESOLVERR_URL,-179/-177 实测)
- **别急着装 Docker / clone 外部项目**:本仓 `tools/flaresolverr_nodriver.py` 就是**免 Docker、FlareSolverr `/v1` 逐字段兼容**的求解端点,用 `nodriver`(uc 现代继任者、驱动真 Chrome)实现。本机 `nodriver 0.50.3 + curl_cffi` 已装,健康检查 + 端到端真解全通过(`SOLVE_OK`)。这是 -145 撞 `undetected-chromedriver 3.5.5` 与现代 Chrome 版本错配时的**零安装、零改码**解。
- **起法(PowerShell,两终端)**:
  ```powershell
  # 终端A —— 起 shim(默认有头,CF 通过率更高;无显示环境才 --headless)
  python tools/flaresolverr_nodriver.py --port 8191
  # 终端B —— 指向它并跑回收
  $env:FLARESOLVERR_URL = "http://127.0.0.1:8191/v1"
  python -m fulltext_fetcher -f <input>.txt -o out\... --email you@org.edu
  ```
- **零改码启用机制**:客户端 `flaresolverr.py::_endpoint()` 只认端点(`cfg.flaresolverr_url` > env `FLARESOLVERR_URL` > 默认 8191,自动补 `/v1`);`download.py::_flaresolverr_enabled()` = `use_flaresolverr` OR `flaresolverr_url` OR env `FLARESOLVERR_URL` 任一为真即启用 → **只 set 一个环境变量就开兜底**,`cli`/`config.py` 不用改。
- **⚠️ 坑1(PowerShell env)**:-177 早稿的 `set FLARESOLVERR_URL=...` 是 **cmd/bat 语法,PowerShell 里静默不生效**(不报错、变量没设上、`_flaserolverr_enabled` 仍 False、兜底静默不触发)。PowerShell 必须 `$env:FLARESOLVERR_URL="..."`;坚持用 cmd 窗口才能 `set`。
- **⚠️ 坑2(端口冲突)**:-145 常驻 FS 可能已占 `8191`,多人同机各用不同端口(`--port 8199`)并相应改 `$env:FLARESOLVERR_URL`。停服务后端口残留:`Get-NetTCPConnection -LocalPort <port> -State Listen` 找 `OwningProcess` → `taskkill /PID <pid> /T /F` 连子树清。
- **⚠️ 坑3(无 CF 小页等满超时)**:对无 CF 页(如 example.com)单次 ~59s(循环到 ~0.9×maxTimeout)。**真实回收不受影响**:shim 只在 `download.py` 已检测 CF 质询后才兜底调用,真 CF 站拿到 `cf_clearance` 提前 break,且**按 origin 缓存(默认 1200s)**→ 80 条 DOI 通常只落 4~5 个域,每域首条付一次解题成本。
- **selftest**:`python -m tools.flaresolverr_nodriver --selftest` → `FLARESOLVERR_NODRIVER_OK`(~21s);`run_all_selftests.py` 默认 **SKIP**(离线全绿 PASS=40 / SKIP=1),`$env:RUN_ONLINE_SELFTESTS="1"` 才真跑联网自检。
- **许可证**:`nodriver = AGPL-3.0`(内部批量回收/个人研究无碍;对外服务须开源或过法务)。
- **变体阶梯(-177)**:仓内 shim(★零安装首选) → **byparr**(Camoufox/Firefox,`pip install uv`+`git clone`+`uv sync`;nodriver headed 仍被特定 Turnstile 识别时上) → 住宅代理(cf_clearance 绑 IP,规模化时)。
- **✅ 纠正 K 节旧结论**:K「batch6 长尾 CF 桶回收杠杆是 FlareSolverr(需 Docker)」里的**「需 Docker」前提已被 -179 推翻**——仓内 nodriver shim **免 Docker** 即可解 CF JS 质询。

### N.2 "过 CF"≠"下到 PDF":三层必须分开看(过 CF / JA3 回放 / 付费墙)

### N.3 【关键发现】ACS 不绑 JA3(回放可过)/ RSC 绑 JA3(回放必 403)—— 分水岭
- **机理**:FS/shim 解 CF 拿 `cf_clearance`+UA 后,`download.py` 把 cookie 交给 **`curl_cffi` 第三方 HTTP 客户端回放**下 PDF。`cf_clearance` 除绑 IP+UA 外,**强 CF 站还绑 JA3/TLS 指纹**。
- **ACS(中等 CF 站,不绑 JA3)→ 回放可过**:实测 `10.1021/acscatal.0c01253`(author-choice OA)FS 真解落 **4.41MB PDF**(`out/recover_b4_cf/`,-145/-149/-179)。
- **RSC(强 CF 站 `pubs.rsc.org`,绑 JA3)→ 回放必 403**:shim 侧 `cf_clearance=YES`,但 `curl_cffi` 回放那步因 **JA3 ≠ 真 Chrome** 仍 403;实测 3 条 RSC `articlepdf` 全 `flaresolverr_failed(cloudflare-challenge http-403)`。**换 byparr(Firefox JA3)/FlareBypasser 同链同理救不了 RSC**(其 JA3 也非回放客户端的 JA3)。
- **一句话**:只要是「solve 拿 cookie → 异客户端回放」这条链,**绑 JA3 的强 CF 站(RSC)必 403**;真解 = 浏览器内直下(N.4)。
- **curl_cffi 背景(-148)**:项目反指纹取回统一走 `curl_cffi`(lexiforest,MIT,`impersonate="chrome"`,与 `scholar.fetcher` 同栈);但它的 JA3 是「curl-impersonate 复刻的 Chrome」,**≠「解题那个真 Chrome 实例」的 JA3**——强 CF 站校验会识破。
- **cookie 回放正确性(中等 CF 站仍适用)**:回传 cookie 直下**必须带 `solve()` 返回的 `user_agent` + 同一出口 IP**,别自己另换 UA;引擎定 UA 家族(shim/FlareBypasser=Chrome UA、byparr=Firefox UA),各自一致、勿混;HTTP/2 站用 `httpx` 而非 `requests`(偶发 403)。

### N.8 CF 过盾根因定论(本波实证闭环,2026-07-02 固化 142;源 -144/-148/-152 实证,-146 授权写锁)

> **一句话**:本波把「CF 过不去」拆成**两条独立代码路径、两个独立根因**,均已实证闭环——(A) FS/shim 路径 0 产出 = **多 shim 重载争用**(非 CF 真破);(B) route-B `render_fetch` 路径卡 CF = **过盾期开了 CDP Fetch/Network 拦截**(非 env/指纹/Chrome 版本)。二者修法各异,勿混。

**#1 FS/shim 路径:0 产出 = 多 shim 重载 thrash(N.7 OSError 的根因定论)**
- 根因:同机并发起了 **7 个 shim(3×flaresolverr + 4×fs_shim_141)**互相 thrash → nodriver 被拖垮 → 全组 CF≈0、solve 报 `OSError:[Errno 22]`(即 N.7 现象)。**非 FS 真挂、非 CF 真破**。实证:-144 kill 掉一个 FS shim 竟释放 ~10 个 chrome 进程。

**#2 route-B 路径:卡 CF 真因 = 过盾期开了 CDP Fetch/Network 拦截(N.4「本机未证」的定论 + 修正)**
- 真因:`render_fetch` route-B 在**过 CF 质询期开着 CDP Fetch 拦截(`fetch.enable`/`network.enable`,render_fetch.py:445-449 导航前)**→ CF 敏感 → 卡 'Just a moment';而 FS shim **全程不开拦截**、只轮询 cf_clearance,故**同机同 Chrome133(甚至 headless)能过**。**指纹 / Chrome 版本 / KeyError 均非 CF 拦路虎**——推翻 N.4「需升 Chrome140+」的旧判。
- **同一根因串联**:N.4 记的 `KeyError:'localNetworkAccessRequestPolicy'` 正是**过盾期开了 CDP Network 域**才触发的 typed 事件崩;不开 Network/Fetch 域即根本走不到那条崩。
- **修法(5 条铁律,-148/-152 实证)**:
  1. 过盾期**绝不开** Fetch/Network 拦截;
  2. 用 **cf_clearance cookie + 可见文案(title / body innerText)** 判过盾——**别**用 HTML 里 `cf-chl`/`challenge-platform` 脚本标记(过盾后残留 → 假阳);
  3. 过盾后**优先 B1**:页内同源 `fetch(pdfUrl,{credentials:'include'}) → FileReader → data-URL`(**不需 `Fetch.enable`**);B1 遇 CSP/跨源失败(`TypeError: Failed to fetch`)才回落 **B2**(过盾后开 `Fetch.enable` RESPONSE + 导航 PDF 直链、网络层 `getResponseBody`)。**拓字节子路径按域分(-152 实测)**:ACS-OA 走 **B1**;**RSC 走 B2**——B1 被 CSP/跨源挡、且 RSC 页无明链 PDF → B2 导航目标用 `publisher_direct._rsc()` 构造 `articlepdf` URL;实现口径=先 B1、遇 CSP/跨源失败回落 B2,B2 的 `Fetch.enable` 只在过盾后开;
  4. **不用** `Page.printToPDF`(重渲染失真、不利 QC);
  5. nodriver `KeyError`(Chrome146 把 `privateNetworkRequestPolicy`→`localNetworkAccessRequestPolicy`)→ `nd.start` 前 **monkeypatch `ClientSecurityState.from_json` 双键兼容**兜底(同 N.4)。

## P. 【重锤】RSC route-B 死结定论:签名 CDN 内联 viewer → 唯「导航后同源页内 fetch」能落字节(方法D,2026-07-02,141;task-d4def0b5)

> **一句话**:RSC(JA3 绑定型 CF)route-B 长期 `no-pdf-captured` 的真因**不是 CF 没过**(CF+SSO 都过了),而是 articlepdf 会 **302 到签名 CDN 直链(`rscj.silverchair-cdn.com/…pdf?Expires=`)并被 Chrome 内联 PDF viewer 打开**:既不触发下载(`set_download_behavior` 无效)、`get_response_body`(Network 域 on_finished + Fetch 域 on_paused,即 -148 的 b2/b2-fetch)也读不到 viewer 流 body → 全 0 字节。**制胜法=导航【落到 CDN .pdf 页后】在其同源上下文 `fetch(location.href)`**(方法D/`b2-viewerfetch`),-141 真机实证 484KB %PDF-1.6。

### P.1 根因三连(-141 探针逐段实证)
- **过盾链路带 SSO 中转**:`https://doi.org/10.1039/d5ra08493h` → CF 质询 → `sso.rsc.org/connect/authorize?client_id=…`(OAuth/SSO 授权回跳)→ 回落 `pubs.rsc.org/ra/article/16/17/15523/…`(真文章页)。**cf_clearance 会在仍停在 SSO 中转页时就置上**——旧 AND 判据(只查 cf_clearance+质询文案)会"偷跑"到 SSO 页、找不到 PDF。
- **articlepdf = 签名 CDN 内联流**:导航 `pubs.rsc.org/en/content/articlepdf/2025/ra/d5ra08493h` → 302 到 `rscj.silverchair-cdn.com/…/d5ra08493h.pdf?Expires=…`,`document.contentType='application/pdf'`,被 Chrome viewer **内联渲染**。故:①无下载事件 → 下载目录空(探针 `saw_partial=False, dir_files=[]`);②Network/Fetch 域 `getResponseBody` 读不到 viewer 流 → -148 的 b2/b2-fetch 必 0 字节(**这就是 -148「Fetch.enable RESPONSE / JA3 B1 fail→b2-fetch」在 RSC 无效的定论**)。
