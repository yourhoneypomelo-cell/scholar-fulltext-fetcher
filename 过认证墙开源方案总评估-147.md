## 〇、TL;DR（决策版）

**一句话**：**本项目现有 route-B 组合（nodriver 真 Chrome + curl_cffi TLS 回放 + FlareSolverr `/v1` 免 Docker nodriver shim + 浏览器内直下字节）已是 2026 开源最优，方向正确、无需换栈**；下一步是**小步升级 + 补合规前置层 + 落 165 行为补丁**，而非引入新框架。

**三堵墙 × 最优解一览**：

| 墙 | 桶（规模） | 门的本质 | 开源最优解 | 净增天花板（诚实） |
|---|---|---|---|---|
| **CF-managed** | ACS 128 / Wiley 23 / AIP 5 / T&F 3 | CF Managed Challenge(403) | nodriver/**zendriver** 过盾 → 不绑 JA3 站 curl_cffi 回放、绑 JA3 站 route-B 内直下 | **个位数**（ACS ~93% 是 SI；Wiley/AIP/T&F 量小） |
| **RSC-JA3** | RSC 金 OA ~8 | **实为伪命题**：RSC 非 passive-JA3 门（是 CF+governor）；"绑 JA3"只在"浏览器解盾→curl 回放异 JA3→403"这条路成立 | route-B **同会话内抓字节**（已在）；真 passive-JA3 长尾站 → curl_cffi `impersonate`（已落地，开 env 即用） | **≈0~低**（TLS 指纹库对 RSC ROI≈0，三源实证） |
| **RSC-governor** | RSC 金 OA ~8（+订阅~59） | 应用层 rate-gate + **坏 reCAPTCHA（不可解）** | **不触发**：165 P1–P3（landing 预热 + per-host 限速/冷却 + 检测）+（可选）住宅 IP | **+0~8**（点估）；订阅~59 **唯 A5 机构订阅** |

**跨方向 6 份报告的 4 条收敛判断**：
1. **nodriver 是 2026 开源天花板**（唯一 0-blocked 28/31），本项目选它是对的；**跨方向一致推荐平滑升级 `zendriver`**（nodriver 活跃分叉）。
2. **★ 关键协同发现**：`zendriver` v0.15.3 **已修 Chrome 146 cookie(`sameParty`缺失)崩溃 + CDP schema 漂移 + headless 过 CF + 官方 Docker**——**正好一并解决 stealth-155 标为"最高 ROI 待盯"的 nodriver Chrome146 cookie 回归 + 本项目已手工 `_patch_nodriver_cdp_compat` 绕开的漂移**。**迁 zendriver = 一步吃两果。**
3. **curl_cffi 主采（已落地为代码）**；**Go 系 TLS（azuretls/utls/tls-client/requests-go）不引入**（Go 栈不适配 requests-only Python）；**Playwright 系（patchright/camoufox/Scrapling）不替换 nodriver**（automation-protocol 第③层更弱，patchright 仅"被迫留 PW 栈"时选）。
4. **开源救不了三件事**（如实认账）：**RSC governor 坏 reCAPTCHA / 数据中心 IP 过 CF / 纯订阅付费墙（唯 A5）**；且**最大净增量来自 A5 机构订阅（路线A），非过认证墙开源工具**（免费/CF 路线已到顶，见《回收实测结论-CF与免费路线到顶》）。

## 三、② Top 3『立即可试』方案（分别对应三堵墙）

### 🥇 墙一 CF-managed（ACS/Wiley/AIP/T&F）→ **验 AIP/T&F 绑 JA3 + nodriver→zendriver + nodriver-cf-verify**
- **立即可试**：① **0 改动探针**——拿 1–2 条 AIP(`pubs.aip.org`)/T&F(`tandfonline.com`) DOI 跑「shim solve→curl_cffi 回放」，看 200/403 决定是否入 `render_fetch._JA3_BOUND_CF_HOSTS` 走 route-B（**本波 ROI 最高的一步**）；② nodriver→zendriver 小样本 A/B（顺带修 Chrome146 cookie+CDP 漂移）；③ 加装 nodriver-cf-verify 处理 Turnstile checkbox。
- **确定性**：高（都在现有 route-B 内小步增量）。

### 🥈 墙二 RSC-JA3（实为"绑 JA3 站下字节"）→ **route-B 同会话抓字节（已在）+ curl_cffi impersonate 验长尾**
- **立即可试**：对**真 passive-JA3 长尾站（非 JS 质询）**开 `FTF_IMPERSONATE_HTTP=1`（curl_cffi impersonate 取回已落地，零新代码）验命中；对 RSC 本身**不指望** TLS 库（三源实证 ROI≈0），维持 route-B `render_download_pdf_bytes` 同会话抓字节。
- **确定性**：中（长尾少量；RSC 走墙三）。

### 🥉 墙三 RSC-governor → **落 165 P1–P3 行为补丁 +（可选）住宅 IP·ProxyLease**
- **立即可试**：先做**零成本、本机直连**的 165 P1–P3（governor 检测 `blocked:rsc-governor` + landing 预热**别怼 articlepdf** + per-host 30–90s 限速/冷却）；验证有效后再挂住宅 IP + `ProxyLease`（IP-会话亲和：浏览器铸 cf_clearance 与 curl 回放**同一住宅出口**）做小样本 A/B。
- **确定性**：中（点估 +0~8；坏 reCAPTCHA 不可解，只能"不触发"）。

## 五-补、谷歌学术可达 PDF · 最快最全程序化获取方案（承接《学术下载器-145》§6.5 · 用户重点③）

> 由学术下载器方向（-145）专供、n7 汇总专设。**核心判断**：「谷歌学术可达的 PDF」绝大多数是 **OA/仓库/预印本**——**用 DOI 走免费 API 多源并发比真爬 Google Scholar 更快更稳**（Scholar 反爬强、需代理归口《代理编排-147》，**只作无 DOI 长尾**）。这与《端到端产品架构差距分析-147》的判断 1「GS 作可选 tier、学术 API 主解析」**独立互证**。

- **最快（DOI 命中即并发免费三件套，取首个 `%PDF` 有效）**：
  - `Semantic Scholar` `openAccessPdf.url`
  - `Unpaywall` `best_oa_location.url_for_pdf`
  - `OpenAlex` `open_access.oa_url`（**2026-02 起需免费 key**；**勿用 lossy 的 `locations[].pdf_url`**）
- **最全（fallback 链）**：`Scholar eprint → S2 → Unpaywall → OpenAlex → PMC/arXiv → FindIt 直链 → route-B`。
- **可直接用**：`paper-search-mcp`（19 源 `download_with_fallback`）、`paper-fetch`（7 源零依赖 + `%PDF` 魔数校验）。
- **吸收整合到本项目**：`scholarly` 取 `eprint_url` + 上述三件套补进 `sources/aggregators.py` / `green_oa.py`；`FindIt`（metapub）出版社直链兜底（延续 `publisher_direct.py` 已有"FindIt 风格"）。
- **与本项目现状对齐**：本项目 `sources/` 已有 aggregators/green_oa/oa_button，三件套是对其"最快并发 + oa_url 口径纠偏（勿用 lossy 字段）"的增强；落地为 OA 层并发取首个有效 PDF，**是端到端演进 P4「GS 专有价值/绿 OA 补漏」的具体配方**。
