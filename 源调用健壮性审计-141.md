## 〇、TL;DR（一页给总指挥 -141）

- **总体结论：核心 13 源健壮性良好，共享 `HttpClient` 是设计到位的统一层**（重试+指数退避+尊重 Retry-After+per-host 限速+Crossref 限速头自适应+SSL 瞬时豁免+主机熔断+curl_cffi 降级，11 场景离线 selftest）。**主要短板在「可观测性」与「无 key 降级的显式化」，不在「会崩」。**
- **三类源**：① **走统一层（健壮）**：unpaywall/openalex/semantic_scholar/crossref/core/openaire + europe_pmc/pmc/biorxiv/doaj/zenodo/hal/arxiv + resolve 解析层（≈13 源）；② **绕统一层自带轻量 HTTP**：wayback（有重试+429退避）、preprints/oa_button（**无重试/无限速/无熔断**，但优雅降级）、websearch（兜底）；③ **异步原型 aio**（httpx，**无重试/限速/熔断**，默认不启用）。
- **头号风险（非崩溃类，但影响判断与召回）**：
  1. **【中·可观测性】静默失败不可区分**——`get_json` 把「网络失败 / 429 耗尽 / 熔断 / 404 / 非 200 / JSON 坏」**全部归一为 None**，各源 `if not data: return []`；pipeline 记的 source 事件 `ok=false` 却 `error=null`，**"源 0 候选"分不清"真无此文"还是"被限速/熔断跳过"** → 规划时易把「临时故障源」误判成「该源到顶/无贡献」。
  2. **【中·无 key 静默降级】** OpenAlex 无 `openalex_key` 只走 mailto 礼貌池（M.4：mailto 2026 收紧 → 持续 429 → `get` 退避耗尽静默返 None），**无显式告警/计数**；S2 无 key 匿名严限；CORE 无 key 直接 `return []`（36M 全文源静默禁用）——**缺 key 用户无感知**。
  3. **【低·限速协调】** `api.crossref.org` 被 **3 处**调用（crossref 源 + resolve DOI 富化/标题反查 + preprints），前两者共享统一层限速，**preprints 自己直连不共享** → 高并发下叠加更易触发 Crossref 429。
- **落地优先级**：**P0**（≈10 行、直接兑现「OpenAlex key 防 429」经验）= 启动期打印 key 状态清单 + 无 openalex_key 显式 warning；**P1** = get_json 区分 None 成因 + 运行末尾汇总各 host 熔断/429 次数 + 对 429 加 host 级冷却；**P2** = preprints 走统一层、Retry-After 补 HTTP-date、oa_button 死源默认下线、aio 补节流。

| 风险 | 级别 | 影响 | 建议（优先级） |
|---|:--:|---|---|
| 静默失败 None 归一（临时故障≈真无数据） | 中 | 误判源到顶、熔断无汇总 | P1 区分成因+末尾统计 |
| 无 key OpenAlex/S2 静默 429 降级 | 中 | 定位主线源失血、召回下降 | **P0 启动告警** + P1 host 冷却 |
| Crossref 三路调用限速不共享（preprints 直连） | 低 | 高并发叠加 429 | P1 preprints 走统一层 |
| core/s2/openalex 缺 key 无启动提示 | 低 | 高价值源静默禁用/降级 | **P0 key 状态清单** |
| Retry-After 只认数字（不认 HTTP-date）；wayback 不读 | 低 | 退避不精准 | P2 补 date 解析 |
| oa_button 死源（410）每条仍发请求 | 低 | 每条一次无效请求 | P2 默认下线/首次410熔断 |
| aio 异步原型无重试/限速/熔断 | 提示 | 生产用会不礼貌 | P2 补节流或标注勿默认 |

|---|---|:--:|

| 退避 | 429/5xx 指数退避 `2^attempt`（上限 30s）；连接错 `2^attempt`（上限 8s） | ✅ |

| 降级 | 重试耗尽返回 `None`（调用方按"无候选"处理）；curl_cffi 缺库降级 requests | ✅ |
| 机构代理 | EZproxy 改写钩子（默认关、OA 域永不改写、零副作用） | ✅ |
| 自检 | `python -m ...http_client` 11 场景离线 selftest | ✅ |

## 三、风险详解（按严重度）

### 中-1 · 静默失败不可区分（可观测性）
- **现象**：`HttpClient.get_json` 中 `r is None`（重试耗尽/熔断）、`status != 200`（404/403/429 耗尽）、`r.json()` 抛错——**三种都返回 None**；上层每个源统一 `if not data: return []`。pipeline `emit("source", ok=bool(cands), error=err)` 的 `err` **只捕获源函数抛异常**，走 get_json→None 的正常路径 `error=None`。
- **后果**：① 事后无法从事件区分"该源真无此 DOI"与"该源被限速/熔断/网络抖动跳过"；② 某 host 熔断后同源后续所有 DOI 静默全 None，**无聚合告警/计数**，可能整批漏召回而不自知；③ **规划 ROI 时把"临时故障源"误判为"该源到顶/无免费贡献"**（直接影响本组"某源是否还值得投入"的判断）。
- **建议（P1）**：`get_json` 返回轻量成因（或经 `ctx.events` emit `rate_limited/circuit_open/http_status/parse_error`）；pipeline source 事件带上 HTTP 层原因；运行末尾在 summary 里汇总「各 host 熔断次数 / 429 次数 / 5xx 次数」。

### 中-2 · 无 key 的 OpenAlex / S2 静默 429 降级（承 OpenAlex key 经验）
- **现象**：`openalex` 无 `openalex_key` → 仅 `mailto`（M.4：mailto 2026-02 政策收紧，礼貌池高并发易 429）；`get` 对 429 退避 4 次后返回 None → 源返 []，**全程无"因 429 降级"的显式日志/计数**。S2 无 `s2_key` → 匿名端点严格限流同理。
- **后果**：OpenAlex 是 resolve **主线定位源**（`resolve_to_paper` DOI 富化优先 OpenAlex），无 key 时静默降级会**系统性拉低元数据富化与候选发现**，且症状隐蔽（表现为"命中率下降"而非报错）。
- **建议**：**P0** 启动期检测并 warning「未设 OPENALEX_KEY，将走礼貌池，高并发易 429 静默降级」；**P1** 对 429 引入 host 级「连续 N 次 429 → 冷却 M 分钟 / 本运行降级跳过」，避免额度耗尽后每条仍空转 4 次退避拖慢整批。

### 低-3 · Crossref 三路调用限速不共享
- **现象**：`api.crossref.org` 被 ①aggregators.Crossref 源 ②resolve 的 DOI 富化 + 标题反查 ③preprints.crossref_preprint_urls 三处调用；①② 走统一层共享 per-host 间隔与自适应头，**③ preprints 自己 `requests.get` 直连、不共享**。
- **后果**：高并发（concurrency 默认 4，可调高）下三路叠加打 Crossref，preprints 那路不受统一节流管辖，更易触发 429。
- **建议（P1）**：preprints 的联网取回改走 `ctx.client`（或注入统一 client），共享限速与自适应头。

### 低-4 · Retry-After 解析局限
- **现象**：统一层 `delay = float(ra) if (ra and ra.isdigit()) else 2**attempt` —— **只认纯数字秒**，RFC 7231 的 HTTP-date 格式（`Wed, 21 Oct 2025 07:28:00 GMT`）会 fallback 到指数退避；wayback 完全不读 Retry-After。
- **建议（P2）**：补 HTTP-date 解析（`email.utils.parsedate_to_datetime`）；wayback 也读 Retry-After。影响小（多数 API 回数字）。

### 低-5 · oa_button 死源仍每条请求
- **现象**：oa.works 2025-11-18 永久停用（410），`OaButtonSource` 仍在 `DEFAULT_SOURCE_ORDER` 中，每条输入发一次注定 410 的请求（虽优雅返 []，但无"本运行已知死→跳过"短路）。
- **建议（P2）**：默认从 `DEFAULT_SOURCE_ORDER` 移除（仅在配置 `oa_button_endpoint` 自建实例时启用），或首次 410 后本运行内熔断跳过。

### 低-6 · 直连源无重试/无限速/无熔断
- **现象**：preprints / oa_button 用裸 `requests.get(timeout=...)`，单次失败即放弃（无退避重试），也不参与统一 per-host 限速与熔断。
- **后果**：调用量小时无碍（preprints 仅标题路径一次、oa_button 已死）；主要问题是与统一层限速状态割裂（见低-3）。
- **建议（P2）**：中长期把 B 类直连源逐步接入统一 client 或共享限速器。

### 提示-7 · aio 异步原型
- **现象**：`aio.py`（httpx 批量定位，C4 路线图）独立于同步核心，**无重试/退避/per-host 限速/熔断**（仅 Semaphore 并发闸 + mailto）；`_httpx_locate_fetch` 单次 `client.get` 查 unpaywall，非 200 → 静默 miss。默认不启用、缺库优雅降级、下载仍走同步核心。
- **建议（P2）**：若要生产化，补每域节流 + 重试；当前保持"实验/勿默认生产"标注即可。

## 五、优点（值得肯定，勿因改进项否定现状）

1. **统一层设计到位**：核心 13 源全走 `HttpClient`，重试/指数退避/per-host 限速/Crossref 限速头自适应/SSL 瞬时豁免/主机熔断/curl_cffi 降级齐备，11 场景离线 selftest 锁定。
2. **SSL 瞬时失败不误熔健康 OA 源**（openaire/zenodo/base/unpaywall 等）——高峰稳定性关键修复。
3. **key 有则用**：openalex/s2 走 premium 池、core 走授权（无 key 优雅跳过不崩）。
4. **全源吞异常不拖垮流水线** + 结构化 `source/download` 事件 + 断点续跑重试白名单。
5. **pipeline 稳健**：arXiv 3s 礼貌间隔、straggler 看门狗（CF 感知放大）、out_dir 并发写锁、收尾必落盘。
6. **直连源也优雅降级**：缺 requests/网络异常/非 200 一律返 []/None，绝不抛。
